# 源表单复核怎么推进

你不需要逐张表从 0 判断。

从现在开始，复核方式改成：

```text
我先给默认结论
你只看哪里不对
你指出例外
我改文件
然后进入下一组
```

也就是说，你不是来做 283 张表的分类员。  
你只负责发现明显不符合业务直觉的地方。

---

## 每轮只看一个小组

不要看整个部门。

每轮只看 5 到 20 张同类表，最多回答一句话：

```text
这组大方向同意。
```

或者：

```text
这组里第 36、37 张果酱房要单独一类。
```

---

## 默认判断规则

如果你不知道怎么判断，就先按下面默认规则走。

| 情况 | 默认结论 |
|---|---|
| 现在还在纸上填 | 保留可填写入口 |
| 多张表只是产品不同、配方不同 | 同类共用页面 |
| 多张表只是区域不同、检查项不同 | 先保留入口，后面再看能不能共用 |
| 表是从库存、批次、主数据自然生成的 | 自动生成或台账视图 |
| 表只是文件、证书、报告、扫描件 | 只当附件 |
| 已经有别的系统覆盖，或者以后不用 | 暂缓或不要做 |

---

## 你只需要关注 4 类例外

看到下面情况再打断我：

1. **不能合并**
   - 例如：看起来像同类表，但实际不同部门、不同场景、不同责任人填写。

2. **不能只是附件**
   - 例如：看起来像文件，但里面有金额、数量、审批、结果，需要结构化记录。

3. **不需要做**
   - 例如：这张表现在已经不用，或已有别的系统覆盖。

4. **填表时没有这个对象**
   - 例如：我写“选择批次”，但实际填表时还没有批次。

---

## 制造部先这样推进

制造部先从第 1 组开始：

`01-制造部.md` 的“第 1 组：投料、配料、打料、面浆、果酱过程”

我给出的默认结论是：

```text
第 1 组默认同类共用页面家族。
```

含义是：

- 每张表都有填写入口。
- 不同产品的投料/配料表，不单独做一套系统。
- 选择产品和配方后，系统带出对应原料行。
- 填的时候可以没有成品批次。
- 先记录日期、时间、线别、产品、工序、班次、人员。
- 包装或入库生成批次后，再关联回去。

你只要判断：

```text
这个默认结论能不能先过？
```

如果你不确定，就先让它过。  
后面看到具体字段或页面时，再改也来得及。

## 你可以直接这样回复

```text
第 1 组先按默认过。
```

或者：

```text
第 1 组默认过，但果酱房单独一类。
```

或者：

```text
第 1 组先别合并，全部保留独立填写。
```

不用解释很多，先把方向推进起来。
