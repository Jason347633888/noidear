## 🎯 使用指南

### 1. Issue 级别说明

本计划采用 **Issue 级别细化**（而非 Task 级别）：

- **Task**: 粗粒度（如"数据库设计"）
- **Issue**: 细粒度（如"设计 RecordTemplate 表字段定义"）

**示例对比**:
```
❌ Task: 实现文档归档功能（20小时）
✅ Issue-P1-1-001: 设计 Document 表归档字段（30分钟）
✅ Issue-P1-1-002: 编写数据库迁移文件（20分钟）
✅ Issue-P1-1-003: 运行迁移并验证（15分钟）
✅ Issue-P1-1-004: 创建 ArchiveDocumentDto（20分钟）
✅ Issue-P1-1-005: 实现 archiveDocument 服务方法（45分钟）
✅ Issue-P1-1-006: 编写 archiveDocument 单元测试（30分钟）
✅ Issue-P1-1-007: 修复测试失败（Debug，30分钟）
... （共 40 个 Issue）
```

### 2. 每个 Issue 包含

```markdown
## Issue-XXX: [Issue 名称]

**类型**: 编码/测试/Debug/修复/验收
**估时**: XX 分钟（精确到 15 分钟）
**依赖**: Issue-YYY（如有）
**优先级**: P0/P1/P2/P3

### 📝 需求描述
[详细描述这个 Issue 要做什么]

### 📂 涉及文件
```
/absolute/path/to/file1.ts
/absolute/path/to/file2.vue
```

### 🔧 开发步骤
- [ ] Step 1: [具体步骤]
- [ ] Step 2: [具体步骤]
- [ ] Step 3: [具体步骤]

### ✅ 验收标准
- [ ] [验收条件1]
- [ ] [验收条件2]

### 🧪 测试清单
- [ ] 单元测试: [测试文件路径]
- [ ] 集成测试: [测试场景]
- [ ] E2E 测试: [用户流程]

### 🐛 预留 Debug 时间
- 预计问题: [可能遇到的问题]
- Debug 预留: XX 分钟

### 🔗 相关 Issue
- 前置: Issue-XXX
- 后续: Issue-YYY
```

### 3. 优先级定义

| 优先级 | 说明 | 时间要求 |
|--------|------|---------|
| **P0** | 阻塞性问题，必须立即完成 | 立即 |
| **P1** | 高优先级 | 本周内 |
| **P2** | 中优先级 | 本月内 |
| **P3** | 低优先级 | 有时间再做 |

### 4. 开发流程

1. **选择 Issue**: 从 P0 开始，按优先级顺序
2. **检查依赖**: 确保前置 Issue 已完成
3. **执行开发步骤**: 按照 🔧 开发步骤逐步执行
4. **运行测试**: 按照 🧪 测试清单验证
5. **预留 Debug**: 遇到问题时使用预留时间
6. **验收确认**: 对照 ✅ 验收标准检查
7. **标记完成**: 勾选所有 checkbox

### 5. 团队协作

- **Issue 分配**: 每个开发者认领 Issue（在 Issue 标题后标注姓名）
- **依赖阻塞**: 如果前置 Issue 未完成，及时沟通
- **时间超支**: 实际时间超过估时 30% 时，更新文档
- **问题记录**: 在 🐛 部分记录实际遇到的问题

## ✅ 使用 TodoList 的最佳实践

### 1. 每日站会
- 每人汇报：昨天完成的 Issue、今天计划的 Issue、遇到的阻塞
- 更新 Issue 状态：✅ 已完成 / 🚧 进行中 / ⏳ 待开始 / ❌ 已阻塞

### 2. 时间跟踪
- 记录每个 Issue 的实际耗时
- 如果实际耗时超过估时 30%，分析原因
- 定期更新估时模型

### 3. 质量保证
- 每个 Issue 完成后，必须通过所有测试
- Code Review：每个编码 Issue 完成后，另一位开发者 Review
- 验收确认：对照 ✅ 验收标准逐项检查

### 4. 文档同步
- 完成一个 TodoList 后，更新 README.md 状态
- 记录遇到的问题和解决方案（补充到 🐛 部分）
- 定期同步到远程仓库
