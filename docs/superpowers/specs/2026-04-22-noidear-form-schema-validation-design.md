# noidear 全量表单体系与校验总方案

## 背景

`noidear` 是主项目，当前已经具备 Vue 3 + Element Plus 前端、NestJS + Prisma 后端、动态模板、任务填报、移动端表单、模板设计器和文档管理能力。现阶段暴露出来的问题不是单一页面缺失，而是整个表单体系缺少统一的语义层和校验层，具体表现为：

- 产品研发主流程的 Step1 只校验了少量字段，和源表单差距很大
- Step2 / Step4 / Step5 / Step6 / Step7 / Step8 的提交校验不完整
- `DynamicForm` 只能处理平铺字段，行级、组合字段、条件字段没有统一接入
- `TableInputField`、`ConstrainedNumberField`、`CheckboxTextField` 没有形成统一的复合校验协议
- 移动端字段类型和校验能力比前端更弱
- 模板设计器可以保存非法配置，导致错误模板进入系统

同时，当前四级记录表单已经扩展到 283 份，字段表达不再是简单输入控件，而是包含表格、默认值、条件显隐、只读带出、审批节点、范围控制、组合规则等表达。继续靠页面特判补洞，只会把问题越积越大。

## 目标

建立一套可持续扩展的全量表单体系，让 `noidear` 满足以下目标：

1. 同一套字段语义能被前端、移动端、模板设计器、后端解析复用
2. 主流程表单能按源表单要求完整校验，不再只拦两个字段
3. 复杂字段类型可以做行级、子项级、组合级校验
4. 条件联动、默认值、只读、枚举、范围规则可以统一表达
5. 模板设计器在保存前就能拦截明显非法配置
6. 现有页面可以渐进接入，不破坏已上线功能

## 要解决的具体问题

本总方案必须覆盖以下问题，不允许只修局部：

### 1. 主流程校验缺失

- Step1 需要从两个字段扩展到源表单要求的完整必填集合
- 申请日期、总经理签名、引入的食品安全危害（含过敏源）等关键规则必须在提交层落地
- Step2 需要补原料行完整性、条件联动、补充说明字段校验
- Step4 需要校验配料行所有必填项，而不是只看批次号
- Step5 / Step6 / Step7 / Step8 不能再直接提交，必须先过校验

### 2. 通用引擎能力不足

- `DynamicForm` 只能做平铺 required / min / max / pattern
- 没有统一的 `validation` 协议
- 行级和子项级错误不能回传到同一错误模型
- `ConstrainedNumberField` 发出的错误没有并入统一校验链
- `CheckboxTextField` 没有勾选后文本必填的组合规则

### 3. 移动端能力不足

- 字段类型覆盖比前端少
- `validateField` 只处理基础值、数字范围、字符串长度
- `FormField` 只渲染基础输入控件
- 表格型字段、审批节点、条件联动字段无法完整表达

### 4. 模板设计器缺少前置校验

- 字段名唯一性没有强校验
- 必填项完整性没有保存前检查
- 枚举选项、默认值、范围规则可能被保存成非法配置
- 错误模板一旦落库，后续页面补救成本很高

## 总体策略

采用“**统一字段规范 + 校验注册表 + 渐进式适配器 + 分阶段落地**”的方案。

### 核心原则

- 先把语义统一，再谈页面补洞
- 先做底座，再做页面特判收敛
- 先比对源表单和字段映射，再落到实现
- 新字段和新规则必须默认进入统一协议
- 旧页面要兼容，但不能继续复制出第二套逻辑

## 方案拆分

### 第一层：字段规范 / DSL

目标是把所有表单字段统一成标准 schema。

开始设计 DSL 前，必须先做一次字段语义比对：

- 从 `04-记录表单` 提取表单编号、表单名称、部门、字段、表格结构、必填规则、范围规则、条件规则
- 从 `SaaS产品构思` 提取规范字段名、实体归属、字段映射、缺口实体、业务语义说明
- 输出差异清单：源表单有但映射缺失、映射存在但源表单已变更、编号或部门归属不一致、字段语义无法自动判断
- 只有确认后的字段语义才能进入 `noidear` 的 DSL 和校验规则

字段至少包含这些语义：

- `fieldType`
- `label`
- `required`
- `defaultValue`
- `options`
- `validation`
- `visibility`
- `readonly`
- `children`
- `rowSchema`
- `dependency`
- `calculation`

字段再往下分四层：

1. 基础元数据：名称、标签、类型、顺序、说明
2. 约束信息：必填、范围、正则、长度、唯一性
3. 交互信息：默认值、只读、条件显隐、联动、自动计算
4. 复合结构：表格、子字段、审批节点、多值组

### 第二层：通用校验引擎

按字段类型注册校验器，而不是把校验散落在各个页面。

建议的校验器类型：

- 文本
- 数字
- 日期 / 日期时间
- 枚举单选
- 枚举多选
- 布尔
- 图片 / 照片
- 表格
- checklist
- 审批签名
- 组合字段

统一错误模型：

- `fieldKey`
- `errorCode`
- `message`
- `severity`
- `rowIndex`
- `childKey`

### 第三层：运行时适配器

把现有组件接入统一协议，但不强推一次性重写。

优先接入：

- `DynamicForm`
- `DynamicField`
- `TableInputField`
- `ConstrainedNumberField`
- `CheckboxTextField`
- 移动端 `FormField`
- 移动端 `validator.ts`
- 模板设计器保存流程

### 第四层：主流程表单补齐

先把最容易出事故、最容易被业务使用的表单补齐：

- 产品开发部 Step1-8
- `新产品开发申请书`
- `产品配方以及工艺参数`
- `产品检验报告`

这些页面要优先做到：

- 提交前完整校验
- 条件字段联动校验
- 数值范围校验
- 多字段组合判断
- 错误定位准确到字段或行

### 第五层：批量表单落地

把 283 份四级表单逐批落到统一 DSL 和统一校验协议里。

做法不是一次性重做全部页面，而是：

- 先按部门批次映射
- 再按缺口字段补规则
- 最后按高频表单优先上线

### 第六层：缺口实体补齐

映射过程中暴露的新字段、新语义、新复合结构，回写到数据模型与实体定义中。

这层只处理必要的实体补齐，不做无边界扩张。

## 推荐实施顺序

这个顺序不要打乱：

1. 字段规范 / DSL
2. 通用校验引擎
3. 主流程 Step1-8
4. `DynamicForm` 与字段组件统一接入
5. 移动端同步接入
6. 模板设计器保存前校验
7. 部门表单批量落地
8. 缺口实体补齐
9. 全库验收

## 阶段定义

### 阶段 1：字段规范 / DSL

输出：

- 标准字段 schema
- 字段类型字典
- `04-记录表单` 与 `SaaS产品构思` 的字段语义差异清单
- 确认后的规范字段名与实体归属
- 条件显隐与默认值语义
- 表格字段和组合字段表达

作用：

- 让后续所有页面和解析器使用同一种字段语言
- 避免 `noidear` 重新发明字段命名和实体归属

### 阶段 2：通用校验引擎

输出：

- 校验注册表
- 错误码规范
- 行级 / 子项级错误回传协议

作用：

- 把规则从页面里抽出去，避免重复堆逻辑

### 阶段 3：主流程 Step1-8

输出：

- 每个 Step 的完整提交校验
- 与源表单一致的必填、范围、组合判断
- 可追踪的错误提示

作用：

- 先把最高频、最高风险的业务链路补牢

### 阶段 4：动态表单组件接入

输出：

- `DynamicForm` 支持复杂 validation
- `TableInputField` 支持行内必填和子项校验
- `ConstrainedNumberField` 接入统一错误链
- `CheckboxTextField` 支持勾选后文本必填

作用：

- 把底层引擎真正接入运行时

### 阶段 5：移动端同步

输出：

- 移动端字段类型补齐
- 统一 validator
- 复杂字段显示与校验接入

作用：

- 避免前后端分裂成两套规则

### 阶段 6：模板设计器校验

输出：

- 保存前唯一性检查
- 枚举与默认值合法性检查
- 必填规则完整性检查

作用：

- 阻止非法模板进入系统

### 阶段 7：部门表单批量落地

输出：

- 各部门字段映射页更新
- 与 `SaaS产品构思` 字段映射的比对结果
- 未覆盖字段清单
- 批量缺口说明

作用：

- 把 283 份表单全部拉进统一体系
- 确认批量导入没有带入旧编号、旧字段名或旧实体归属

### 阶段 8：缺口实体补齐

输出：

- 新实体定义
- 字段归属说明
- 与既有实体的关联关系

作用：

- 补齐映射过程中暴露的结构性空洞

### 阶段 9：全库验收

输出：

- 编号一致性检查
- 校验覆盖率检查
- 残留旧逻辑清单
- 回归测试结果

作用：

- 确认 `noidear` 已经能承载全量表单体系

## 风险与约束

- 不允许把所有逻辑继续堆在单个大组件里
- 不允许前端和移动端各写一套不同语义
- 不允许先做业务页再补底座
- 不允许跳过 `04-记录表单` 与 `SaaS产品构思` 的字段映射比对
- 不允许破坏现有模板和旧记录的读取
- 不允许为了快而跳过模板设计器校验

## 验收标准

1. `Step1` 能按源表单要求完整校验，不再只拦两个字段
2. `Step2`、`Step4`、`Step5`、`Step6`、`Step7`、`Step8` 都有提交前校验
3. `DynamicForm` 可以处理复杂字段类型和行级错误
4. 移动端和前端输出一致的校验结果
5. 模板设计器能拦截明显非法配置
6. 283 份四级表单可逐批映射到统一规则
7. 缺口实体可回写，不丢语义
8. `04-记录表单`、`SaaS产品构思`、`noidear` 三者的表单编号、字段语义、实体归属有可追踪比对结果
9. 现有功能不回归

## 交付方式

这个总方案不会一次性改完所有页面，而是按阶段拆成独立任务包执行。

每个任务包都要满足：

- 边界清晰
- 改动文件尽量少
- 先写测试或至少补回归检查
- 完成后单独验证
