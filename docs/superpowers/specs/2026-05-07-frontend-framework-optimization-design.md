# 全局框架层前端优化设计

- 日期：2026-05-07
- 适用范围：`client`
- phase：Phase 1
- 当前样板：`/dashboard`

## 1. 背景

当前前端已经覆盖大量业务入口，但页面体验仍然以“模块堆叠式后台”为主：

- 左侧导航入口多、层级深，但一级分组缺少稳定优先级，高频执行入口与低频治理入口并列出现。
- 顶栏、页面标题区、卡片、列表、表单在不同页面中各自成型，缺少统一的骨架和状态语言。
- 首页仍偏“系统概览”，没有体现当前用户“今天应该先处理什么”的执行视角。
- 视觉上已有一些品牌化尝试，但落点分散，导致全站在高级感、密度和一致性之间失衡。

本次优化不新增业务入口，不做“新门户替代旧入口”的扩张，而是对现有入口做重组、下沉、收口和样式统一。

## 2. 目标

### 2.1 产品目标

将前端从“功能模块集合”提升为“高级型运营后台”：

- 保持工具型系统的效率和可扫描性。
- 强化品牌感、视觉层级和页面秩序。
- 让用户在现有入口内更快找到高频任务，不依赖新增快捷页或临时入口。

### 2.2 体验目标

- 高频执行路径优先于低频治理路径。
- 页面骨架统一，用户切换模块时不需要重新学习布局。
- 风险、截止时间、状态变化可被快速识别。
- 桌面端先打透，移动端在 Phase 1 中只定义明确的折叠与降级规则。

### 2.3 非目标

- 不新增一级导航入口。
- 不删除任何现有业务入口。
- 不在 Phase 1 重构所有业务页。
- 不引入营销页、说明页、装饰性大屏或无业务价值的中间页。

## 3. 范围

### 3.1 包含范围

Phase 1 覆盖以下层级：

1. 全局框架：`Layout`、侧边导航、顶栏、主内容容器、页面标题区、面包屑呈现。
2. 全局设计系统：颜色、字体层级、间距、圆角、阴影、状态色、空状态。
3. 全局组件模式：按钮、tag、panel、table、filter bar、form section、操作栏。
4. 首批验证页面：
   - `/dashboard`
   - `/my-todos`

### 3.2 排除范围

- 文控中心业务结构本身的重定义。
- 追溯页面的领域交互重设计。
- 生产、仓储、文控各模块的全量逐页改造。
- 服务端接口新增或领域建模变更。

## 4. 设计原则

### 4.1 入口不扩张

所有优化必须基于现有入口完成。允许：

- 调整分组归属
- 调整一级/二级层级
- 改造入口命名呈现和视觉权重

不允许：

- 新增临时首页入口
- 为绕开原有问题再造一个平行页面
- 用“快捷功能”替代原本应该被优化的正式入口

### 4.2 执行优先

界面排序原则：

1. 当前任务
2. 到期/逾期事项
3. 风险和异常
4. 协同与历史
5. 治理与配置

### 4.3 高级型运营后台

视觉方向应满足：

- 有品牌感和层次，但不营销化。
- 有足够留白，但不牺牲信息密度。
- 强调专业、秩序和运营感，而不是消费产品式轻快卡片感。

### 4.4 风险色节制使用

风险色仅服务于：

- 逾期
- 高风险
- 异常
- 需立即处理的状态

禁止把风险色扩散成全站主色或大面积背景色。

### 4.5 桌面优先，移动可折叠

Phase 1 主战场是桌面端。移动端要求：

- 不出现内容重叠或结构崩坏。
- 明确哪些区域折叠、隐藏、纵向堆叠。
- 不在本阶段为所有复杂表格单独设计移动专属交互。

## 5. 信息架构策略

## 5.1 导航重组规则

现有导航保留全部入口，但执行以下策略：

1. 高频入口前置：与当前用户日常操作强相关的页面优先进入一级可见区域。
2. 低频入口下沉：治理、配置、统计、备份、导入导出等低频项下沉到二级层级。
3. 同链路聚合：同一业务链条中的列表、详情入口、配置入口在导航结构上保持邻近。
4. 分组语义稳定：一级分组必须表达用户任务场景，而不是随意按技术模块拆分。

## 5.2 一级分组建议

Phase 1 spec 中建议导航按以下方向重组：

1. 工作执行
2. 文控与审批
3. 生产与批次
4. 质量与合规
5. 设备与现场
6. 统计与追溯
7. 系统管理

说明：

- 这是重组现有入口的归类建议，不代表新增模块。
- 低频入口仍存在，但不再与高频执行入口共享同等视觉权重。

## 6. 全局框架设计

## 6.1 侧边导航

目标：

- 更强分组边界
- 更轻默认噪音
- 更明确当前所在链路

规则：

- 一级导航作为“场景分组”，默认展示组标题、主入口、展开箭头。
- 二级导航承载低频页和同链路补充页。
- 当前激活页不仅高亮自身，还要让所属分组保持明确选中状态。
- 收起状态保留图标与 tooltip，不展示过多文字噪音。

## 6.2 顶栏

顶栏只保留全局能力：

- 导航展开/收起
- 当前页面定位信息
- 全局通知
- 用户菜单

禁止在顶栏继续堆放页面级过滤器和模块级按钮。

## 6.3 页面标题区

每个页面统一使用标题区骨架：

- 主标题
- 一行简短上下文说明
- 右侧主操作区或筛选摘要

不同页面不再各自发明“横幅、卡片头、双层标题、随机副标题”等结构。

## 6.4 主内容容器

主内容容器统一采用：

- 稳定最大宽度
- 明确左右边距
- 标准 section 间距
- 面板与内容带状布局并存

目标是让列表页、工作台页、详情页在骨架上能互相识别为同一系统。

## 7. 视觉系统

## 7.1 色彩方向

主方向：

- 暖灰/浅米色背景
- 深墨色正文
- 稳定的中性色边界
- 节制的品牌强调色

语义色：

- 主强调：用于主按钮、当前选中、重点动作
- 成功：完成、已通过、已发布
- 警告：到期、待处理、存在阻塞风险
- 危险：逾期、高风险、失败、异常

## 7.2 字体层级

推荐层级：

- 页面主标题：更有识别度的中文 serif 或带个性的标题字族
- 面板标题：稳定、清晰、偏工具化 sans
- 表格/表单/正文：高可读 sans

要求：

- 不使用过度花哨字体
- 不依赖超大字营造层次
- 所有标题和正文必须服务扫描效率

## 7.3 面板与卡片

Phase 1 不再广泛使用“到处都是白卡片”的默认后台写法。

改为：

- 页面级 section 使用无框布局或轻面板
- 只有需要强调、聚焦或隔离内容时才使用 panel/card
- card 圆角、阴影、边框应统一，不允许不同页面自由发挥

## 8. 组件模式

## 8.1 按钮

- 主按钮：只用于当前页面最关键动作
- 次按钮：用于辅助操作
- text 按钮：只用于轻跳转和二级操作
- 图标按钮：只放全局或表格级工具动作

禁止同一视图出现多个争夺注意力的主按钮。

## 8.2 Tag / 状态标识

Tag 主要表达：

- 类型
- 状态
- 风险等级

要求：

- 小尺寸
- 高对比可读
- 颜色语义固定
- 不作为装饰存在

## 8.3 表格

列表页应采用更紧凑但更清晰的密度策略：

- 控制列数优先级
- 关键列左侧固定在视觉焦点
- 状态、风险、截止时间以结构化方式呈现
- 行 hover 和操作区明确但不过度跳跃

## 8.4 表单

表单统一按 section 分组：

- 基础信息
- 关联对象
- 状态/风险
- 备注与附件

不再允许长表单以无分组方式一路堆到底。

## 8.5 空状态

空状态应表达：

- 当前没有数据
- 用户下一步可做什么

不能只显示“暂无数据”后结束。

## 9. 首批验证页

## 9.1 `/dashboard`

定位：

- 执行门户
- 今日任务总览
- 到期优先、风险次之

要求：

- 作为 Phase 1 的视觉与信息骨架样板页
- 不再作为泛系统概览页
- 不新增新的业务入口，只重构已有入口的呈现方式

当前 demo 价值：

- 已验证“执行门户”方向可行
- 可作为后续抽全局规则的参照
- 不应停留在单页特例，后续必须抽象为全局框架能力

## 9.2 `/my-todos`

定位：

- `dashboard` 主队列的完整展开页

要求：

- 与 dashboard 使用同一任务语言
- 排序、tag、风险、截止时间语义一致
- 保持更高密度的列表处理体验

## 10. 响应式规则

## 10.1 桌面端

- 左侧导航常驻
- 内容区采用双栏或多区布局
- 页面标题区与主要操作并列展示

## 10.2 平板与窄桌面

- 次要侧栏下移
- 二级信息块纵向堆叠
- 列表仍优先保留关键信息列

## 10.3 移动端

Phase 1 只定义以下降级：

- 导航抽屉化
- 双栏变单栏
- 辅助信息区下移
- 操作按钮横向改纵向或底部堆叠

复杂数据表格允许在移动端退化为信息卡片或关键字段子集，但此设计不在本 phase 落地。

## 11. 实施顺序

建议按以下顺序执行：

1. 抽全局 token 和基础视觉变量
2. 重构 `Layout`
3. 统一页面标题区和主内容容器
4. 重做 `/dashboard`
5. 重做 `/my-todos`
6. 评估是否进入 Phase 2：
   - 文控链
   - 生产追溯链
   - 质量合规链

## 12. 验证标准

Phase 1 完成后，应满足以下标准：

1. 用户不需要新增学习新入口，只在现有入口中获得更顺的路径。
2. 高频入口更靠前，低频入口不再干扰主流程。
3. `dashboard` 与 `my-todos` 在视觉和信息语言上明显属于同一体系。
4. 桌面端体验完整，移动端至少不崩坏且折叠逻辑明确。
5. 不引入新的“快捷页”“中转页”“伪入口”。

## 13. 风险与约束

### 13.1 风险

- 如果先改页面不改框架，后续会返工。
- 如果一次性尝试覆盖所有业务页，spec 会失控且实现会发散。
- 如果导航只做美化不做分组重排，执行效率提升有限。

### 13.2 约束

- 现有全量 `client build:check` 存在与本任务无关的基线 TS 问题，验证时需区分“全局基线问题”和“本次改动引入问题”。
- 本次优化以客户端框架与现有入口呈现优化为主，不应借机扩大为领域功能重构。

## 14. 结论

Phase 1 应采用“框架先行”的推进方式：

- 先统一骨架与信息架构
- 再用 `dashboard` 和 `my-todos` 做验证落地
- 确认稳定后，再逐链路扩展到其它现有入口

这是在“不新增无效入口，只优化现有入口”的前提下，最稳且返工最少的路径。
