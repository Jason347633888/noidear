# 2026-04-26 Document Control Operations Center Design

## 1. Goal

Extend the phase-1 document-control center into a food-safety compliance operations center.

Phase 2 establishes the controlled file system for `01-06`, generic references, the `04` record-form landing index, and the basic workbench. This phase builds on that foundation to turn controlled files into operational workflows:

- read confirmation
- training needs from changed files
- internal-audit coverage
- CAPA/change/recall impact analysis
- external requirement impact analysis
- document health scoring
- audit-chain evidence navigation

The product position is:

> The operations center uses controlled-document relationships to drive and verify food-safety system execution. It still does not own completed business records or replace the business modules that generate evidence.

## 2. Preconditions

This design assumes the phase-1 document-control center is implemented:

- `Document` has document type, source folder, metadata, tags, review due date, external expiry date, and lineage fields.
- `DocumentReference` can reference documents, record templates, record lists, business modules, business objects, company files, and external files.
- `RecordFormLandingEntry` maps `04-и®°еЅ•иЎЁеЌ•` source forms to their target modules, models, templates, and routes.
- The document-control workbench can detect due review, expiring external files, obsolete references, broken entrances, missing landing targets, and missing metadata.
- Existing modules continue to own their facts: training owns training projects and learning records; internal audit owns plans, findings, and reports; CAPA owns corrective actions; change modules own change events and verification; traceability owns recall/traceability query results.

If any of these prerequisites are missing, implementation must first complete or adapt the phase-1 document-control center instead of creating a parallel operations layer.

## 3. Scope

### 3.1 In Scope

This phase includes:

- document read-confirmation requirements by role, department, or user
- training-needs suggestions generated by new or revised effective documents
- internal-audit coverage view based on document references and audit findings
- CAPA/change/recall impact analysis over document references and target modules
- external file impact analysis for laws, standards, customer requirements, and supplier/third-party files
- document-control health dashboard
- audit-chain view from requirement to document to SOP to record entrance to business evidence

### 3.2 Out Of Scope

This phase must not:

- auto-create high-stakes CAPA, recall, or audit records without explicit user action
- auto-submit messages, notifications, or regulatory communications to third parties
- duplicate training, internal audit, CAPA, change, or traceability data into `Document`
- create a second workflow engine
- reclassify the 283 source forms
- replace target-module permissions
- enforce opaque AI compliance scoring as a decision source

## 4. Core Operating Concepts

### 4.1 Document Requirement

A document requirement defines who must acknowledge or act on a document after publication or revision.

Examples:

- All QA users must read `CX-11 ж ‡иЇ†е’ЊеЏЇиїЅжєЇжЂ§жЋ§е€¶зЁ‹еєЏ`.
- Production supervisors must read a revised cleaning SOP.
- Procurement users must read a changed supplier-control procedure.

Requirements are not training records. They are the bridge between a controlled document and downstream read confirmation or training suggestions.

### 4.2 Training Need

A training need is a recommendation or draft planning item caused by a document becoming effective or changing materially.

It should link to the existing training module rather than storing training completion inside the document module.

States:

- `suggested`
- `accepted`
- `dismissed`
- `linked`

### 4.3 Audit Coverage

Audit coverage answers whether controlled documents are covered by internal-audit plans, findings, and reports.

The document module should not own internal-audit findings. It should show references into the internal-audit module and identify gaps such as:

- effective procedure has no audit coverage in the selected period
- SOP has findings but no linked CAPA
- external requirement is not linked to any procedure

### 4.4 Impact Analysis

Impact analysis answers what may be affected by a document, external file, change, CAPA, recall, or obsolete reference.

Impact is derived from:

- `DocumentReference`
- document type and metadata
- record-form landing entries
- linked business modules and routes
- existing business objects in target modules when available

It is a decision-support view. It should not mutate business modules unless the user explicitly chooses an action.

### 4.5 Audit Chain

An audit chain is a readable path:

```text
External requirement or system clause
  -> procedure
  -> SOP / work instruction
  -> record-form landing entry or business module
  -> completed evidence in owning module
  -> finding / CAPA / verification when present
```

The chain uses links and snapshots for navigation; the owning modules remain authoritative.

## 5. Product Surfaces

### 5.1 Read Confirmation Center

Shows:

- required readers by department, role, and user
- confirmation status
- overdue readers
- documents with no requirement configured
- documents revised after prior confirmation

Actions:

- configure requirement
- confirm read
- export confirmation list
- jump to document detail

### 5.2 Training Need Center

Shows:

- documents requiring training review
- suggested training needs
- accepted and dismissed suggestions
- linked training projects

Actions:

- accept suggestion and link to a training project
- dismiss with reason
- jump to training module

The first implementation should create suggestions, not auto-create training projects.

### 5.3 Audit Coverage Center

Shows:

- procedures and SOPs covered by internal-audit plans
- documents with audit findings
- documents with unresolved findings
- documents with findings but no linked CAPA
- external requirements without document coverage

Actions:

- filter by audit period
- jump to internal-audit plan, finding, or report
- mark a coverage item as reviewed

### 5.4 Impact Analysis Workbench

Supports impact queries for:

- document revision
- obsolete document
- external file expiry or replacement
- change event
- CAPA
- recall or traceability event

Outputs:

- impacted documents
- impacted SOPs
- impacted record-form entries
- impacted business-module entrances
- impacted departments
- suggested follow-up actions

Actions remain manual and explicit.

### 5.5 Document Health Dashboard

Shows health metrics:

- effective documents missing owner or review date
- overdue review count
- expired external file count
- obsolete references count
- broken entrance count
- read-confirmation overdue count
- training suggestion backlog
- audit coverage gaps

Health scores should be transparent rule-based counts, not opaque AI scores.

## 6. Data Model

Recommended new models:

### 6.1 DocumentReadRequirement

Purpose: configure who must read a document.

Fields:

- `id`
- `documentId`
- `scopeType`: `department | role | user`
- `scopeId`
- `requiredBy`
- `requiredAt`
- `dueAt`
- `reason`
- `status`: `active | closed`

### 6.2 DocumentTrainingNeed

Purpose: store training suggestions caused by document publication or revision.

Fields:

- `id`
- `documentId`
- `triggerType`: `new_effective_document | revised_document | external_requirement_change | manual`
- `targetDepartment`
- `targetRole`
- `reason`
- `status`: `suggested | accepted | dismissed | linked`
- `linkedTrainingProjectId`
- `dismissedReason`
- `createdBy`
- `createdAt`
- `updatedAt`

### 6.3 DocumentImpactReview

Purpose: store manual impact-review sessions and their reviewed status.

Fields:

- `id`
- `sourceType`: `document | external_file | change_event | corrective_action | recall | traceability`
- `sourceId`
- `title`
- `status`: `draft | reviewed | closed`
- `summary`
- `reviewedBy`
- `reviewedAt`
- `createdAt`
- `updatedAt`

### 6.4 DocumentImpactItem

Purpose: store impact-review items derived from references.

Fields:

- `id`
- `reviewId`
- `targetType`
- `targetId`
- `targetRoute`
- `targetLabel`
- `relationType`
- `impactLevel`: `low | medium | high`
- `suggestedAction`
- `status`: `open | accepted | dismissed | done`
- `notes`

### 6.5 DocumentCoverageReview

Purpose: optionally store audit-coverage review snapshots.

Fields:

- `id`
- `periodStart`
- `periodEnd`
- `documentId`
- `coverageStatus`: `covered | gap | not_applicable`

- `findingId`

- `reviewedAt`

## 7. Services

### 7.1 Read Requirement Service

Responsibilities:

- create and update read requirements
- compute required readers from user, department, and role scopes
- compare requirements with existing read confirmations
- return overdue and missing confirmations

### 7.2 Training Need Service

Responsibilities:

- create suggestions when documents become effective or are revised
- deduplicate suggestions by document and target scope
- allow accept, dismiss, and link actions
- expose route links to the training module

### 7.3 Coverage Service

Responsibilities:

- read document references and internal-audit data
- identify coverage gaps
- link documents to audit plans, findings, reports, and CAPA when references exist
- generate a coverage summary for a period

### 7.4 Impact Service

Responsibilities:

- build impact candidates from references and landing entries
- create impact-review sessions
- store reviewed impact items
- avoid writing to target modules without explicit user action

### 7.5 Health Service

Responsibilities:

- aggregate transparent counts from document control, read requirements, training needs, audit coverage, and references
- return health sections and drill-down routes

### 7.6 Audit Chain Service

Responsibilities:

- traverse references from a document or external requirement
- include record-form landing entries and target routes
- include available evidence references without copying evidence data
- return expandable chains for the UI

## 9. Permissions

Use existing auth and target-module permissions.

Recommended permissions:

- `document:read_requirement_manage`
- `document:training_need_manage`
- `document:audit_coverage_view`
- `document:impact_review_manage`
- `document:health_view`
- `document:audit_chain_view`

Rules:

- ordinary users can confirm their own reads
- document-control admins manage requirements and impact reviews
- training admins accept or link training suggestions
- audit users view audit coverage according to audit-module permissions
- target-module routes must still enforce target-module permissions

## 10. Events

Use existing document lifecycle events where possible.

Events:

- `document.published`
- `document.updated`
- `document.obsoleted`
- `document.external.expiring`
- `document.reference.changed`

Event reactions:

- effective document creates read-requirement status rows or read-status projections
- effective or revised document creates training suggestions when configured
- obsolete document triggers impact candidates
- external file expiry triggers impact candidates

The first implementation can generate suggestions on demand or through service calls. It does not need a background event bus beyond existing Nest event emitter usage.

## 11. Error Handling

Required behavior:

- missing document: return 404
- unsupported scope type: return 400
- duplicate active read requirement: return conflict
- accepting dismissed training need: return conflict
- linking training need without project id: return 400
- impact review source missing: return 404
- impact item target route missing: keep item open and mark `suggestedAction`
- user lacks target-module permission: show restricted route label, not raw data

## 12. Testing Strategy

Tests must prove:

- read requirements calculate missing and overdue confirmations
- training needs deduplicate by document and target scope
- audit coverage finds covered and uncovered documents
- impact service derives candidates from references and landing entries
- health service returns transparent counts
- audit-chain service traverses references without copying target evidence
- frontend pages render counts and route links

## 13. Success Criteria

The operations center is successful when users can:

- see who still needs to read a changed effective document
- turn a document revision into a training need without manually searching affected users
- see which procedures/SOPs have internal-audit coverage gaps
- understand which documents, SOPs, forms, and modules are affected by a changed external requirement
- view document-control health with drill-downs
- show an auditor a clear chain from requirement to procedure to SOP to record entrance to business evidence

## 14. Relationship To Phase 2

This phase must be implemented after phase 2 or behind guards that check for phase-2 fields and endpoints.

Do not change the phase-2 responsibility split:

- document module owns controlled-document operations and references
- dynamic record layer owns dynamic records
- independent business modules own structured facts
- model-landing remains the authority for how 283 source forms land
