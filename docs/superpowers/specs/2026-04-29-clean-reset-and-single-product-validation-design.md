# 清库保留边界与单产品真实实测设计

## 1. 背景

当前系统已经完成文控中心、记录表单治理、产品主数据联动、变更主线、文控工作台、证据链、Markdown wikilink 等多轮建设。下一步不适合继续堆演示数据，而应把现有混杂数据清理到一个可控状态，再用一个真实产品跑完整业务链路。

本设计只定义清理边界、保留对象和第一轮真实实测数据方案，不直接执行删库。

## 2. 目标

1. 清理当前库中所有演示、测试、临时业务数据。
2. 保留文控中心所有体系文件、文档版本、文档引用和文控治理配置。
3. 保留设备台账，包括普通设备台账和计量/监测设备台账。
4. 保留账号、组织、权限、菜单、记录模板、审批定义等系统基础配置。
5. 用一个真实产品重新建立最小但完整的业务闭环，用于手工实测。

## 3. 不做范围

本阶段不做以下事情：

1. 不改数据库 schema。
2. 不重构业务模块。
3. 不删除文控文档文件本体。
4. 不删除设备台账。
5. 不一次性录入全部产品、全部供应商、全部历史批次。
6. 不用假数据覆盖真实业务判断。

## 4. 清理策略

采用“保留体系骨架，清空业务流水，重建一个真实产品链路”的策略。

### 4.1 必须保留

以下数据必须保留：

| 类别 | 典型模型 | 说明 |
| --- | --- | --- |
| 用户与权限 | User, Role, FineGrainedPermission, Department | 登录、授权、组织关系不能清 |
| 文控文档 | Document, DocumentVersion, DocumentReference | 文控中心所有文件、版本、引用都保留 |
| 文控治理配置 | DocumentReadRequirement, DocumentReadConfirmation, DocumentTrainingNeed, DocumentCoverageReview, RecordFormLandingEntry | 用于继续验证文控闭环 |
| 记录表单模板 | RecordTemplate | 表单模板是系统配置，不是业务流水 |
| 审批/流程定义 | ApprovalDefinition, ProcessTemplate 等定义类模型 | 定义保留，实例可清 |
| 设备台账 | Equipment | 保留设备基础档案 |
| 计量设备台账 | MeasuringEquipment | 保留监视测量设备基础档案 |
| 基础系统配置 | 菜单、系统参数、种子配置 | 保证系统可登录、可导航、可授权 |

### 4.2 可以清理

以下数据可以清理：

| 类别 | 典型模型 | 说明 |
| --- | --- | --- |
| 产品业务数据 | Product, Recipe, RecipeLine, ProcessStep | 重新用一个真实产品建立 |
| 供应链业务数据 | Supplier, Material, MaterialBatch, IncomingInspection, StockRecord | 如现有数据混杂测试数据，则清空重建 |
| 仓储业务流水 | MaterialInbound, MaterialRequisition, MaterialReturn, MaterialScrap, StockCount | 属于业务事件 |
| 生产业务流水 | ProductionBatch, BatchMaterialUsage, FinishedGoodsBatch, ProductionRun | 清空后重建一个真实批次 |
| 质量/过程记录 | CCPRecord, EnvironmentRecord, ProcessMonitorRecord, MetalDetectionLog, ReworkRecord, Sample | 随真实产品链路重新产生 |
| 发货/客户链路 | Customer, DeliveryNote | 第一轮可选，如要测正追到客户再建 |
| 变更治理实例 | ChangeEvent, ChangeImpactAssessment, ChangeVerificationRecord, ChangeApproval, ChangeEventFormTask | 定义保留，实例清空 |
| 动态填写记录 | Record, RecordChangeLog, DeviationReport | 历史填写记录清空，模板保留 |
| 待办与审批实例 | TodoTask, ApprovalInstance, ApprovalTask, ApprovalAction | 运行实例清空 |
| 测试报表/导出/快照 | TraceabilitySnapshot 等 | 清空后重新生成 |

### 4.3 清理前安全要求

清理前必须完成：

1. 导出数据库备份。
2. 导出文件存储目录清单，确认文控文件路径仍存在。
3. 生成保留表数量快照。
4. 生成待清理表数量快照。
5. 先在本地或测试库执行，不直接对生产库执行。
6. 清理后跑登录、文控中心、设备台账三个冒烟检查。

## 5. 第一轮真实产品

第一轮建议选择：

**香蕉蒸蛋糕（原味）**

选择原因：

1. 属于典型食品生产链路，能覆盖产品、配方、物料、批次、投料、过程、检验和追溯。
2. 源表单中已有相关生产记录、投料记录和过程记录语义。
3. 单产品范围足够小，适合先验证系统链路，不会陷入全量资料录入。

如实际业务上当前最常生产的是其他产品，可以替换为一个同等复杂度的真实产品，但第一轮必须只选一个。

## 6. 第一轮需要准备的真实数据

### 6.1 产品主数据

至少准备：

- 产品编号
- 产品名称
- 规格
- 净含量
- 保质期
- 储存方式
- 食用方式
- 执行标准
- 标签过敏原
- 产品类型

### 6.2 物料主数据

至少准备 5 到 8 个关键物料：

- 鸡蛋
- 小麦粉
- 白砂糖
- 食用油
- 香蕉浆或香蕉粉
- 包装膜
- 脱氧剂或其他关键包材

每个物料至少有：

- 物料编号
- 物料名称
- 规格
- 单位
- 类别
- 保质期
- 是否过敏原
- 状态

### 6.3 供应商

每个关键物料至少关联一个供应商。

供应商至少有：

- 供应商编号
- 供应商名称
- 状态
- 联系方式
- 资质文件或资质有效期

### 6.4 物料批次

每个关键物料至少建立一个真实批次。

物料批次至少有：

- 系统批次号
- 供应商批号
- 生产日期
- 到期日期
- 入库数量
- 库位
- 供应商
- 当前状态

### 6.5 配方

为该产品建立一个 active 配方。

配方必须：

- 引用 Product
- 配方明细引用 Material
- 关键物料标记为关键项
- 如已有工序区域，则配方行关联 WorkshopArea

### 6.6 生产批次

建立一个真实生产批次，例如：

`PB-20260429-001`

生产批次至少有：

- 批次号
- 产品
- 配方
- 计划产量
- 实际产量
- 生产日期
- 班次
- 生产线
- 状态

### 6.7 投料记录

用 `BatchMaterialUsage` 把生产批次和物料批次连接起来。

这是追溯链最关键的数据，不允许只在备注或动态表单里写物料名称。

### 6.8 过程与质量记录

第一轮至少准备：

- 过程参数记录
- 金属探测记录
- CCP 或关键控制点记录
- 成品首检或放行记录
- 留样记录

如果某些表单尚未有独立业务表，则先通过 `RecordTemplate + Record` 填写，但必须通过 `productionBatchId`、`relatedBatchId` 或 `entity_links` 关联到生产批次。

### 6.9 发货链路

第一轮可选。

如果要验证正向追溯到客户，则准备：

- 一个客户
- 一个成品批次
- 一条发货记录

如果当前重点是生产和文控闭环，发货链路可以放到第二轮。

## 7. 第一轮实测路径

按以下顺序测试：

1. 登录系统。
2. 打开文控中心，确认体系文件仍在。
3. 打开设备台账，确认设备仍在。
4. 建立或确认产品主数据。
5. 建立物料和供应商。
6. 建立物料批次和库存。
7. 建立产品配方。
8. 建立生产批次。
9. 录入投料。
10. 录入过程、金检、CCP、检验、留样记录。
11. 查询生产批次追溯。
12. 从文控文件查看引用、关联记录和证据链。
13. 检查文控工作台是否能显示真实问题。

## 8. 验收标准

清理后必须满足：

1. 文控中心文档数量不减少。
2. 文控文档文件可打开。
3. 文档引用健康状态可查询。
4. 设备台账数量不减少。
5. 用户可正常登录。
6. 记录模板仍可查看。
7. 业务流水数据已清空。
8. 一个真实产品可以完成从产品、配方、物料批次、生产批次、投料到追溯查询的链路。
9. 追溯查询能从生产批次反查到物料批次和供应商。
10. 记录表单填写后能挂到产品、生产批次或变更事件，而不是孤立记录。

## 9. 风险与控制

| 风险 | 控制方式 |
| --- | --- |
| 误删文控文档 | 清理脚本必须显式排除 Document 及文控关联表 |
| 误删设备台账 | 清理脚本必须显式排除 Equipment 和 MeasuringEquipment |
| 外键删除顺序错误 | 先生成依赖顺序，使用事务或分阶段删除 |
| 模板被误删 | RecordTemplate 不清，Record 清 |
| 真实产品链路断裂 | 第一轮只做一个产品，逐步补齐 Product -> Recipe -> MaterialLot -> ProductionBatch -> BatchMaterialUsage |
| 文件存储和数据库不一致 | 清理前后检查文档 filePath 是否存在 |

## 10. 推荐下一步

下一步写执行计划，计划应包含：

1. 数据库备份步骤。
2. 保留表/清理表清单。
3. 清理 SQL 或脚本的执行顺序。
4. 清理前后数量快照。
5. 单产品真实数据录入清单。
6. 手工实测脚本。
7. 回滚方案。

执行计划完成并确认后，才允许写清理脚本或实际执行数据清理。
