# 车间配料区投料与产品批次归集设计

## 1. 背景

现有系统已经合并产品主数据、配方、配料区域和产品批次链路，但正式投料入口仍存在一个业务缺口：投料弹窗里的原辅料批次仍接近手填逻辑，且默认假设投料时已经知道产品批次。

现场实际不是这样。配料发生在车间配料区，包装喷码和产品入库发生在后段；最终产品批次通常由包装机喷码时确定。夜班 `20:00-08:00` 会跨自然日，同一个班可能产生两个日期产品批次。一次配料也可能跨多个产品批次，不适合在配料当下强行绑定 `ProductionBatch`。

本设计把正式投料入口调整到车间配料区，先形成真实配料投入池，包装入库后再把投入池归集到包装确认出来的产品批次。

## 2. 设计目标

- 配料区作为正式投料入口。
- 原辅料批次由系统按配料区库存和 FIFO 自动推荐，不让用户普通手填批次 ID。
- 允许现场按实际情况调整批次或数量，但必须填写原因并留痕。
- 配料当下不强制绑定产品批次。
- 包装/入库环节按现场喷码人工录入产品批次号并确认产品批次。
- 支持一次配料记录关联多个产品批次。
- 追溯展示真实投入池和相关原辅料批次，不伪造每个产品批次精确用了多少 kg 原料。
- 班前、班后、接班盘点形成闭环；盘点差异属于异常，不作为日常盘盈盘亏处理。

## 3. 当前范围边界

本设计只处理“真实配料区投料 -> 包装喷码/入库确认产品批次 -> 配料投入池归集 -> 追溯查询”的闭环。

本设计暂不处理提前生产未来日期导致的双生产数据问题，包括：

- 实际产品批次与标识产品批次拆分。
- 未来喷码日期与真实生产日期的映射。
- 提前生产审批、销售需求绑定、客户特殊日期要求。
- 按实际生产口径和标识喷码口径同时做物料平衡。

在本设计范围内，业务文档统一使用“产品批次”。代码当前由 `ProductionBatch` 承载产品批次，`ProductionBatch.batchNumber` 就是现场包装喷码确认的产品批次号。后续如果要支持提前生产未来日期，应另开独立设计，把两套生产数据和批次映射关系单独建模，不能混入本次配料区投料闭环。

本设计不再把 `FinishedGoodsBatch` 作为独立业务批次概念。当前代码中的 `FinishedGoodsBatch` 属于早期通用追溯模型遗留的成品库存扩展层，后续实施计划应安排迁移或剔除，避免形成“生产批次、喷码批次、成品批次”三套概念。

## 4. 现有可复用模型

本设计不新造一套平行库存或追溯事实源。优先复用以下对象：

| 对象 | 当前用途 | 本设计中的定位 |
| --- | --- | --- |
| `MaterialBatch` | 原辅料批次 | 原辅料真实批号和来源 |
| `StagingAreaStock` | 车间暂存区库存 | 配料区按批次结存 |
| `StagingAreaRecord` | 暂存区盘点记录 | 扩展为班前、班后、接班盘点 |
| `StagingAreaTransfer` | 暂存区迁移记录 | 配料区之间或仓库到配料区移动留痕 |
| `Recipe` / `RecipeLine` | 产品配方 | 配方明细和所属配料区 |
| `WorkshopArea` | 配料区域 | 公司级配料区主数据 |
| `ProductionBatch` | 产品批次 | 包装喷码/入库后确认的唯一业务批次 |
| `BatchMaterialUsage` | 旧投料追溯事实表 | 保留兼容旧链路，新链路不依赖其精确分摊 |

## 5. 新增核心对象

### 5.1 班组与排班

`Team` 表示责任班组，例如 A 组、B 组、C 组。

建议字段：

- `id`
- `name`
- `leaderId`
- `status`
- `createdAt`
- `updatedAt`

`TeamMember` 记录班组成员。

建议字段：

- `id`
- `teamId`
- `employeeId`
- `role`
- `joinedAt`
- `leftAt`

`ShiftType` 表示时间班次，例如日班和夜班。

建议字段：

- `id`
- `name`
- `startTime`
- `endTime`
- `crossesDay`

默认规则：

- 日班：`08:00-20:00`
- 夜班：`20:00-08:00`

`TeamShiftSchedule` 表示排班表。

建议字段：

- `id`
- `workDate`
- `shiftTypeId`
- `teamId`
- `leaderId`
- `createdById`
- `updatedById`

包装/入库确认批次时，系统根据入库时间判断日班/夜班，再根据排班表带出责任班组和负责人。现场允许修改，但必须记录原值、新值、修改人、修改时间和原因。

### 5.2 配料执行记录

`MixingExecution` 表示一次配料执行，也就是一个投入池。

建议字段：

- `id`
- `workDate`
- `shiftTypeId`
- `teamId`
- `leaderId`
- `areaId`
- `areaNameSnapshot`
- `recipeId`
- `productId`
- `sequenceNo`
- `startedAt`
- `endedAt`
- `status`
- `operatorId`
- `remark`

`MixingExecutionLine` 表示一次配料中的某个原辅料批次投入。

建议字段：

- `id`
- `mixingExecutionId`
- `recipeLineId`
- `materialId`
- `materialNameSnapshot`
- `materialBatchId`
- `materialBatchNoSnapshot`
- `recommendedQuantity`
- `actualQuantity`
- `unit`
- `fifoOrder`
- `isManualAdjusted`
- `adjustReason`
- `operatorId`
- `createdAt`

配料记录是新链路中的真实投料事实源。它记录实际投入了哪些原辅料批次、多少数量、谁操作、何时操作、是否人工调整。

### 5.3 产品批次配料归集

`BatchMixingAggregation` 表示产品批次和配料投入池之间的归集关系。

建议字段：

- `id`
- `productionBatchId`
- `mixingExecutionId`
- `confirmedById`
- `confirmedAt`
- `confirmRemark`

该关系是多对多：

- 一个配料执行记录可关联多个产品批次。
- 一个产品批次可关联多个配料执行记录。

归集不要求填写每个产品批次分摊了多少原料重量。

## 6. 业务流程

### 6.1 开班与接班盘点

配料区开班前必须完成盘点。

盘点粒度必须是：

```text
配料区 + 物料 + 原辅料批次
```

不能只盘某个物料的总量，否则 FIFO 会失真。

规则：

- 班前盘点不一致：禁止开班配料。
- 接班盘点不一致：禁止开班配料。
- 班后盘点不一致：不能正常交班。
- 差异必须进入异常处理，不能当作普通盘盈盘亏跳过。

差异处理方式：

- 校正库存：适用于历史录入错误或确认以实物为准的情况。
- 补录投料：适用于现场已投料但系统漏登记。
- 退料/转回仓库：适用于现场物料退回仓库或转出配料区。
- 报废/损耗：适用于洒落、污染、设备异常导致损耗。
- 仅记录说明：适用于差异还在调查或待管理层确认。

所有处理方式都必须记录差异物料、批次、账面数量、实盘数量、差异数量、原因、处理方式、处理人、确认人和时间。

### 6.2 配料区 FIFO 投料

配料区员工进入车间配料区工作台后，只能看到自己区域的任务、库存和盘点状态。

投料时系统按以下规则推荐原辅料批次：

1. 根据 `RecipeLine.areaId` 找到当前配料区需要配的物料。
2. 查询该配料区 `StagingAreaStock` 中该物料的可用批次。
3. 按 FIFO 排序，优先旧批次。
4. 旧批次库存足够时，默认扣旧批次。
5. 旧批次库存不足时，自动拆到新批次。

示例：

```text
配方需要白砂糖 30kg

小料房库存：
- 白砂糖 A批：剩余 12kg，入区时间最早
- 白砂糖 B批：剩余 50kg

系统默认生成：
- A批 12kg
- B批 18kg
```

如果现场发现 A 批实际只剩 10kg，可调整为：

```text
- A批 10kg
- B批 20kg
原因：A批现场实物不足，按实际称量调整
```

人工调整必须留痕。系统应记录原推荐方案、调整后方案、调整人、调整时间和原因。

投料扣减的是 `StagingAreaStock`，不是仓库总库存。仓库总库存应在领料/调拨到配料区时扣减。

### 6.3 包装喷码与入库产品批次确认

产品批次在包装喷码/装箱入库环节确认。

包装负责人或包装区域领班录入：

- 产品
- 产品批次号
- 包装/入库时间
- 入库数量
- 单位
- 包装机
- 包装线
- 备注

系统根据时间自动带出：

- 时间班次：日班/夜班
- 责任班组
- 负责人

产品批次号由现场人工录入。系统不强制按规则生成产品批次号，因为批次号以现场包装机实际打印为准。

批次展示建议：

```text
产品：香蕉蒸蛋糕
产品批次号：20260429
时间班次：夜班
责任班组：C组
负责人：李四
包装机：1号包装机
生产线：1号线
入库时间：2026-04-29 23:40
入库数量：按实际入库单位显示，例如 120 箱 / 2880 件
```

包装机是追溯记录字段，不参与产品批次身份唯一性。只要记录了包装机，就能反推出生产线。

### 6.4 配料归集

配料归集由两段确认完成：

1. 包装/入库负责人确认产出了哪些产品批次、入库了多少。
2. 生产班长/负责人确认这些产品批次关联了哪些配料投入池。

归集的含义是：

```text
把某段时间内、某些配料区做过的配料投入记录，
关联到后面包装喷码确认出来的一个或多个产品批次。
```

归集不是分摊原料重量。

示例：

```text
夜班 23:30，小料房第 N 次配料
夜班 23:35，筛粉间第 N 次配料
夜班 23:40，称油间第 N 次配料

00:00 前包装出来一批：产品批次号 20260429
00:00 后包装出来一批：产品批次号 20260430
```

系统允许：

```text
产品批次 20260429 关联：
- 小料房 23:30 第 N 次配料
- 筛粉间 23:35 第 N 次配料
- 称油间 23:40 第 N 次配料

产品批次 20260430 也关联：
- 小料房 23:30 第 N 次配料
- 筛粉间 23:35 第 N 次配料
- 称油间 23:40 第 N 次配料
```

系统不要求填写：

```text
20260429 精确用了其中 42kg
20260430 精确用了其中 58kg
```

因为现场无法真实判断这种精确分摊。伪造精确分摊会让追溯数据看起来准确，实际失真。

### 6.5 追溯查询

新链路追溯查询直接走：

```text
ProductionBatch
 -> BatchMixingAggregation
 -> MixingExecution
 -> MixingExecutionLine
 -> MaterialBatch
```

追溯结果展示三层：

1. 产品批次层：产品、产品批次号、日/夜班、责任班组、负责人、包装机、入库数量。
2. 投入池层：关联的配料执行记录、配料区、第几次配料、操作人、时间。
3. 原辅料层：实际使用的原辅料批次、投入量、是否人工调整、调整原因。

`BatchMaterialUsage` 保留旧链路兼容。新配料区投料闭环不再强行生成每个 `ProductionBatch` 的精确原料用量记录。

### 6.6 物料平衡

物料平衡不等于批次精确分摊。

物料平衡计算：

- 配料投入总量
- 成品入库数量/重量
- 报废
- 留样
- 退料
- 损耗
- 产率
- 异常说明

产率可能低于 100%，也可能高于 100%。例如蒸蛋糕加工时吸收生产用水，而生产用水未计入原辅料配料总量，导致成品重量可能超过原辅料投入重量。

因此系统不应以“投入总量必须等于产出分摊总量”为硬校验。

## 7. 页面设计

### 7.1 班组与排班

功能：

- 维护班组。
- 维护班组长和成员。
- 维护日班/夜班时间规则。
- 维护每日排班。
- 支持临时换班，换班必须留原因。

### 7.2 车间配料区工作台

这是正式投料入口。

功能：

- 按配料区过滤任务和库存。
- 显示班前/接班盘点状态。
- 未完成盘点或盘点异常时禁用配料。
- 按配方明细生成配料任务。
- 按区域库存和 FIFO 推荐原辅料批次。
- 支持跨批次扣减。
- 支持人工调整并强制填写原因。
- 生成 `MixingExecution` 和 `MixingExecutionLine`。

### 7.3 包装/入库批次确认

功能：

- 选择产品。
- 录入产品批次号。
- 录入入库数量。
- 选择包装机。
- 系统自动带出日/夜班、责任班组和负责人。
- 允许修改班组和负责人，但必须留原因。
- 生成或确认 `ProductionBatch`。

### 7.4 配料归集工作台

功能：

- 显示未归集配料执行记录。
- 显示待归集产品批次。
- 系统按产品、配方和时间窗口推荐可能关联关系。
- 生产班长确认归集。
- 支持一个配料执行记录关联多个产品批次。
- 支持一个产品批次关联多个配料执行记录。
- 显示异常提示：未归集配料、无配料批次、时间明显不合理、盘点未闭环。

### 7.5 盘点与异常处理

功能：

- 班前盘点。
- 班后盘点。
- 接班盘点。
- 差异异常创建。
- 差异原因和处理方式选择。
- 处理结果确认。
- 支持按班次、配料区、物料批次倒查投料记录和人工调整记录。

## 8. 后端服务边界

建议模块边界：

| 模块 | 职责 |
| --- | --- |
| `team` / `shift` | 班组、成员、日夜班规则、排班 |
| `warehouse/staging-area` | 配料区库存、调拨、盘点、差异处理 |
| `mixing` | 配料执行、FIFO 推荐、实际投料扣减、人工调整留痕 |
| `packaging` / `product-batch` | 包装喷码产品批次确认、入库数量、包装机、责任班组 |
| `traceability` | 新追溯链查询 |

`batch-trace` 中现有的 `MaterialUsageService` 可保留兼容，但新配料闭环不应继续把它作为唯一入口。

## 9. 关键校验

### 9.1 开班校验

- 当前配料区必须完成班前或接班盘点。
- 盘点差异未处理时禁止开班。
- 当前时间必须能匹配日班/夜班。
- 排班表缺失时允许人工选择班组，但必须留原因。

### 9.2 配料校验

- 只能从当前配料区 `StagingAreaStock` 扣减。
- 默认 FIFO 推荐老批次。
- 老批次不足时自动拆批。
- 人工改批次/数量必须填写原因。
- 实际投料数量必须大于 0。
- 投料完成后必须扣减配料区库存。

### 9.3 包装入库校验

- 产品必选。
- 产品批次号必填。
- 入库数量必须大于 0。
- 包装机必选。
- 责任班组和负责人必须确定。
- 包装机不参与产品批次唯一性，但必须作为追溯字段保存。

### 9.4 归集校验

- 一个产品批次必须至少关联一组配料投入池，才能视为追溯完整。
- 一个配料投入池可以关联多个产品批次。
- 不要求原料重量分摊。
- 归集确认必须记录确认人和确认时间。
- 时间明显冲突时提示。例如配料结束时间明显晚于包装结束时间。

### 9.5 交接校验

- 班后盘点不一致时，不能正常交班。
- 接班盘点不一致时，不能正常开班。
- 差异异常必须处理或明确进入待确认状态。

## 10. 测试策略

### 10.1 单元测试

- FIFO 推荐老批次优先。
- 老批次不足时自动拆到新批次。
- 人工调整必须有原因。
- 班前盘点不一致禁止开班。
- 配料扣减只扣 `StagingAreaStock`。
- 归集支持多对多。
- 归集不生成虚假分摊重量。

### 10.2 集成测试

- 领料到配料区 -> 班前盘点 -> 配料 -> 包装入库 -> 归集 -> 追溯查询。
- 夜班跨日期：同一班次产生两个产品批次。
- 一次配料关联两个产品批次。
- 班后盘点差异 -> 异常处理 -> 交班完成。
- 接班盘点差异 -> 禁止开班 -> 处理后允许开班。

### 10.3 前端测试

- 配料区入口不能普通手填原辅料批次 ID。
- FIFO 推荐批次可见。
- 人工调整原因必填。
- 未完成班前盘点时禁用配料按钮。
- 归集工作台能显示未归集配料和待确认批次。
- 包装入库能按时间带出日/夜班和责任班组。

## 11. 不做事项

本设计不做以下事情：

- 不把生产计划当成最终产品批次事实源。
- 不要求配料当下必须填写产品批次。
- 不按比例估算每个产品批次精确用了多少 kg 原辅料。
- 不把包装机作为产品批次身份的一部分。
- 不把盘点差异当成日常普通盘盈盘亏。
- 不处理提前生产未来日期、两套生产数据、实际产品批次与标识产品批次映射。
- 不把 `FinishedGoodsBatch` 作为新的业务批次层继续扩展。
- 不新建平行的原辅料主数据、产品主数据或库存主数据。

## 12. 成功标准

- 配料区能按 FIFO 从本区域库存选择并扣减原辅料批次。
- 配料执行记录可以在不知道产品批次的情况下保存。
- 包装入库后能按现场喷码确认产品批次。
- 产品批次能归集到一个或多个配料投入池。
- 追溯查询能从产品批次追到相关配料执行记录和原辅料批次。
- 班前、班后、接班盘点能阻断异常流程。
- 差异异常能留下处理原因和处理结果。
