# 产品信息工作台与变更联动设计

## 背景

当前产品研发相关入口分散在“产品目录”“配方管理”“工序步骤管理”“研发流程列表”等页面中。配方和工序步骤虽然在数据模型上都归属于产品/配方链路，但用户可以从独立页面直接编辑，容易绕过变更申请、审批和历史版本管理。

体系文件中心也存在入口体验问题：进入文件库时直接展示所有文件，用户没有先从左侧文件夹选择范围，导致文件库像全量列表而不是受控文件夹浏览。

## 目标

1. 将产品研发的日常工作入口收敛到“产品信息”工作台。
2. 将配方、历史/归档配方、工序步骤、CCP 参数放到产品详情中管理。
3. 复用现有 `ChangeEvent` 模块，不新建“配方变更申请”模块。
4. 配方、工艺步骤、CCP/关键限值、炉温等修改必须先有变更申请。
5. 一个已批准的配方或工艺变更单只能执行一次落版动作，不能一对多复用。
6. 体系文件中心默认不展示全部文件，必须先选择左侧文件夹。

## 非目标

1. 本设计不重新设计产品规格书。产品规格书模板/落库问题后续单独核对。
2. 本设计不重建变更管理、审批中心或研发流程模块。
3. 本设计不改变食品安全主数据口径：产品仍为 `Product`，配方仍为 `Recipe + RecipeLine`，工序仍为 `ProcessStep`。

## 入口与菜单

产品研发菜单保留两个主要入口：

- 产品信息
- 研发流程列表

普通用户菜单移除或隐藏：

- 配方管理
- 工序步骤管理

配方管理和工序步骤管理不删除能力，而是收敛为产品详情页中的子区块。必要时可以保留管理员只读台账入口，用于全局排查和数据维护，但不作为日常编辑入口。

体系文件中心文件库入口调整为：

- 初始状态右侧显示空状态：“请选择左侧文件夹查看文件”。
- 左侧文件夹被选择后才查询并展示文件。
- 搜索默认限定在已选文件夹范围内。
- 如后续需要全库搜索，应显式提供“全库搜索”入口，不作为默认行为。

## 产品信息详情页

产品信息页从“产品列表 + 编辑弹窗”升级为“产品列表 + 产品详情页”。

产品列表保留：

- 查看详情
- 发起新产品研发
- 历史产品建档
- 归档产品

产品详情页包含以下区块：

### 产品基础信息

展示产品编号、名称、状态、来源、创建时间等主数据字段。这里只处理产品主数据本身，不处理配方和工艺修改。

### 当前配方

展示当前 active 配方版本、物料行、配料区域、关键物料。

动作规则：

- “查看配方”始终可用。
- “发起配方变更”创建现有 `ChangeEvent`，不创建新模块。
- “执行配方修改”仅在存在已批准且未执行的 `ChangeEvent(change_type = recipe)` 时显示。

### 工艺步骤与 CCP 参数

展示该产品或当前配方下的工序步骤、炉温、风机、关键限值、监控方法、监控频率和纠正措施。

动作规则：

- “查看工艺步骤”始终可用。
- “发起工艺/CCP 变更”创建现有 `ChangeEvent`。
- “执行工艺修改”仅在存在已批准且未执行的 `ChangeEvent(change_type = process 或 haccp)` 时显示。

### 历史版本

展示历史/归档配方和历史工艺记录。每条历史记录应显示版本号、状态、来源变更单和查看入口。

配方删除语义统一为归档，不做物理删除。归档配方从当前配方视图隐藏，但在历史版本区块可查看。

### 关联变更记录

展示与该产品、当前配方、历史配方、工序步骤相关的 `ChangeEvent`。状态分为：

- 草稿/审批中：继续处理
- 已批准未执行：可执行
- 已执行：只读查看执行结果
- 已拒绝：只读查看

## 变更模块复用

现有 `ChangeEvent` 是本设计唯一的变更申请来源。产品详情页不创建新的配方变更模块，只调用现有变更能力。

发起配方变更时创建：

- `change_type = recipe`
- `relations` 至少关联当前 `Product`
- 如已有当前配方，同时关联当前 `Recipe`

发起工艺或 CCP 变更时创建：

- `change_type = process` 或 `haccp`
- `relations` 关联当前 `Product`
- 如变更针对特定配方或工序，同时关联 `Recipe` 或 `ProcessStep`

变更申请创建后继续复用现有能力：

- 默认表单任务
- 合规评估
- 验证记录
- 统一审批
- 变更详情页

## 一对一执行规则

变更申请和落版动作必须一对一。

### 配方变更

1. 用户从产品详情发起配方变更。
2. 系统创建 `ChangeEvent(change_type = recipe)` 并关联产品和当前配方。
3. 变更走现有审批/验证流程。
4. 审批通过后，产品详情显示“执行配方修改”。
5. 用户进入配方编辑页时必须带 `changeEventId`。
6. 保存时创建新的 `Recipe` 版本，旧 active 配方归档。
7. 新 `Recipe` 记录来源 `changeEventId`。
8. `ChangeEvent` 标记为已执行，并记录执行出的 `Recipe`。
9. 已执行的 `ChangeEvent` 不能再次进入配方编辑或再次生成配方版本。

### 工艺/CCP 变更

工艺步骤、CCP、关键限值、炉温、监控频率和纠正措施变更遵循同样规则：

1. 先创建 `ChangeEvent(change_type = process 或 haccp)`。
2. 审批通过后才允许执行修改。
3. 执行修改必须绑定 `changeEventId`。
4. 执行后标记变更单已执行，不能重复使用。

## 数据与状态要求

实现时需要能表达以下关系：

- 新配方版本来源于哪个 `ChangeEvent`。
- 新工艺或工序版本来源于哪个 `ChangeEvent`。
- 某个 `ChangeEvent` 是否已经执行过。
- 某个 `ChangeEvent` 执行生成了哪个业务对象。

推荐语义字段：

- `Recipe.changeEventId`
- 工艺版本或 `ProcessStep` 的 `changeEventId`
- `ChangeEvent.executedAt`
- `ChangeEvent.executedResourceType`
- `ChangeEvent.executedResourceId`

字段名称可按现有 schema 风格调整，但语义必须保留。

## 用户路径

### 配方修改

1. 打开产品信息。
2. 进入产品详情。
3. 点击“发起配方变更”。
4. 进入现有变更详情页处理申请、评估、验证和审批。
5. 审批通过后回到产品详情。
6. 点击“执行配方修改”。
7. 配方编辑页加载当前配方，并携带 `changeEventId`。
8. 保存后生成新配方版本、归档旧版本，并标记变更已执行。

### 工艺修改

1. 打开产品详情。
2. 在“工艺步骤与 CCP 参数”区块点击“发起工艺/CCP 变更”。
3. 走现有 `ChangeEvent` 流程。
4. 审批通过后点击“执行工艺修改”。
5. 保存后生成或更新受控工艺版本，并标记变更已执行。

## 验证要求

实现后至少验证：

1. 体系文件中心初始不加载全部文档。
2. 产品研发菜单不再暴露普通用户用的独立配方管理和工序步骤管理入口。
3. 产品详情能展示当前配方、历史/归档配方、工序步骤和关联变更。
4. 未审批的配方变更不能执行配方修改。
5. 已批准未执行的配方变更可以执行一次配方修改。
6. 已执行的配方变更不能再次生成新配方版本。
7. 工艺/CCP 变更遵循同样的一对一执行约束。
8. 配方归档后从当前视图隐藏，但历史版本仍可查看。
