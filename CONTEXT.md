# Noidear 食品安全 SaaS 上下文

本上下文记录食品安全 SaaS 中主数据、文控、记录表单、生产、配料、库存和追溯相关的业务语言。它用于避免在设计、实现和评审中把同一个现场概念拆成多个平行概念。

本文分两层：

- **Stable Domain Language**：稳定业务术语和已确认领域边界，默认可作为后续讨论口径。
- **Implementation Notes**：当前代码状态快照，必须按标注的核验日期和证据重新确认，不能当作永久事实。

## Stable Domain Language

### 主数据与产品链

**产品档案**:
产品进入系统后的统一主数据，承载产品编号、名称、状态和后续生产引用身份。
_Avoid_: 历史产品表、研发产品表、产品名称事实源

**产品编号**:
系统生成、用户可见的产品业务编号，用于搜索、打印和业务沟通。
_Avoid_: 手填最终编号、另造 productNo/productNumber

**产品名称快照**:
业务记录在发生当时保存的产品名称展示值，用于历史还原，不作为产品事实源。
_Avoid_: 用快照反向维护产品档案

**产品归档**:
产品退出正常业务但保留历史生产、投料、发货、投诉和追溯证据的状态。
_Avoid_: 删除产品、移入回收站

**产品画像**:
围绕一个产品聚合展示标签、配方、过敏原、风险区、保质期、规格书和验证摘要的产品视图或轻量元数据对象。它用于统一查看和导航，不复制产品档案、配方、检验或研发流程中的源头字段。
_Avoid_: 把产品画像做成第二张产品宽表、在画像里手填标签/过敏原/规格书事实

**产品标签画像**:
由产品档案、配方、物料过敏原、产品过敏原、企业资质和法规字段聚合生成的标签信息视图。
_Avoid_: 独立手填产品标签表、让标签字段和配方/过敏原事实源分叉

**产品规格书**:
面向客户、审核或内部放行使用的产品资料导出快照，由产品、配方、工艺、标签、检验和证据模板生成。
_Avoid_: 把规格书当成日常录入主表、在规格书里维护产品和配方事实

**物料过敏原画像**:
针对单一物料或某供应商物料的过敏原事实源，记录该物料含有、可能含有或交叉接触的过敏原信息。
_Avoid_: 只在产品过敏原清单里手填原料过敏原

**产品过敏原画像**:
由当前生效配方、配方明细和物料过敏原画像自动汇总生成的产品级过敏原结论，可物化保存结果和版本，但不得作为人工手填源头。
_Avoid_: 手工维护产品过敏原清单、配方变更后忘记同步过敏原

**产品风险区判定**:
产品开发阶段对产品适用高风险区、高关注区、常温高关注区等风险分类的判定记录。
_Avoid_: 把风险区判定做成每批生产或每次检验都要填写的记录

**产品验证记录**:
产品开发、配方变更或工艺变更后，为确认产品、配方或工艺可行而形成的验证事件，可引用检验记录、变更事件和证据文件。
_Avoid_: 把验证记录塞进三级文件正文、或用验证记录替代检验记录本身

**配方**:
某个产品在一个版本下的物料组成和理论用量。
_Avoid_: Formula、工艺参数表和配方并列使用

**配方明细**:
配方中某一种物料的理论用量、单位、关键性和所属配料区域。
_Avoid_: 在投料记录里重新定义理论配方

**配方归档**:
配方版本退出新生产使用但保留历史解释能力的状态。
_Avoid_: 删除配方、覆盖旧配方

**工序**:
产品或配方下的生产步骤，承载过程控制、CCP 和检验要求的业务位置。
_Avoid_: 把工序当成普通表单章节

### 批次、配料与库存

**生产计划**:
对未来某日或某班次要生产哪些产品、数量和产线安排的计划事实。
_Avoid_: 用产品批次、生产运行或审批任务替代生产计划

**生产计划明细**:
生产计划中针对某个产品、规格、数量、配方或产线的一条计划行。
_Avoid_: 从产品批次字段反推计划明细

**现场执行任务**:
由生产计划明细派生给现场人员执行的生产动作要求，例如配料、检查、包装确认或入库确认。
_Avoid_: 用审批任务、动态表单任务或产品批次替代现场执行任务

**生产运行**:
某条产线、某个班次或某段时间实际运行某个产品的现场运行上下文。
_Avoid_: 用生产运行替代生产计划、现场执行任务或产品批次

**产品批次**:
包装机喷码、成品入库、发货、投诉、召回和追溯共同使用的产品批次。
_Avoid_: 生产批次、喷码批次、成品批次并列使用

**产品批次号**:
产品批次的现场编号，来自包装机实际喷码或包装/入库环节确认。
_Avoid_: 另造一套成品批次号

**产品批次确认**:
确认某个产品批次号已经在现场出现，可以开始归集包装、入库、检验、异常和追溯事实的业务动作。它只表示“这个批号开始了”，不表示成品已经入库，也不表示这个批号已经结束。
_Avoid_: 把产品批次确认等同于成品入库确认或批次关闭确认

**成品入库确认**:
仓库或成品暂存区接收人确认接收某个产品批次成品的库存动作。它属于现有 **仓库模块** 的库存流水能力扩展，不是新建平行的成品入库模块、包装入库模块或第二套仓库模块。实现时应复用现有 `WarehouseModule` / `InventoryMovement` / `InventoryMovementLedgerService` 口径，为产品批次补 `object_type = production_batch`、`movement_type = production_in` 的记录能力。它是生成成品入库 **库存流水**（`InventoryMovement.object_type = production_batch` 且 `movement_type = production_in`）的唯一业务动作；同一个 **产品批次** 在关闭前可以分多次成品入库，每次确认各生成一条 `production_in` 库存流水并指向同一个产品批次；包装确认只记录包装/喷码/数量等生产事实，不直接写成品库存流水。当前不新增独立 **包装确认** 或 **包装产出记录** 对象，也不以包装累计数量作为入库数量门禁；入库数量以仓库或成品暂存区接收确认为准。入库确认只能选择或确认已有 **产品批次**，不得修改产品批次号；首发不新增现场实物批号扫描、录入或自动比对功能。如果现场人工发现批号不一致，按线下流程退回生产/包装侧重新确认或另行异常处理。
_Avoid_: 包装确认时提前写 production_in、把包装完成等同于成品已入库、由包装员工代替仓库/成品暂存区接收人确认入库、绕过入库确认直接增加库存余额、新增 PackagingConfirmation/PackagingOutputRecord、新建平行成品入库模块、新造第二套仓库库存流水、用不可得的包装累计数量卡成品入库、入库时修改产品批次号、用仓库侧新批号覆盖包装/喷码批号

**批次关闭确认**:
确认某个产品批次不会再继续产生包装或成品入库事实的轻量状态动作。连续生产时通常由切换到下一批号触发上一批关闭确认；非连续生产时由现场在停机、结束生产或无下一批号时关闭当前批号。关闭前必须已经存在该产品批次至少一条成品入库 **库存流水**（`InventoryMovement.object_type = production_batch` 且 `movement_type = production_in`），但不是只能有一条；它只负责让 `ProductionBatch.status` 进入 `completed`，不保存关闭人、关闭时间、关闭原因、实际产量快照或入库量快照；入库量以 **库存流水** 为准。
_Avoid_: 没有成品入库库存流水就关闭产品批次、completed 后继续追加普通 production_in 入库流水、投料完成/生产运行结束/质检完成后自动把产品批次改成 completed、为关闭动作新增一套关闭快照字段

**配料执行**:
某个配料区在某个时间完成的一次配料活动。
_Avoid_: 把一次配料直接叫成产品批次

**投料记录**:
产品批次与原辅料批次之间用于追溯的桥接结果，由配料执行明细确认后自动归账生成。
_Avoid_: 把投料记录当成配料区现场唯一录入入口、手工补填追溯桥

**配料投入池**:
一次或多次配料执行形成的实际投入集合，可通过执行归集关系关联到一个或多个产品批次。
_Avoid_: 虚构每个产品批次精确分摊的原料重量、把配料投入池当成逐物料批次用量明细

**原辅料批次**:
原料、辅料或包材进入库存后的批次，投料时按配料区库存和 FIFO 使用。
_Avoid_: 手填无法关联库存的批次文本

**来料登记**:
供应商送货到厂后对物料、数量、供应商批号和收货信息的登记事实，尚不代表物料已经可用。
_Avoid_: 登记后直接形成可用原辅料批次

**来料检验门禁**:
决定一条来料明细是否可以生成可用原辅料批次和库存流水的最终质量判定。
_Avoid_: 把来料检验当成已存在批次的事后补充记录

**检验标准**:
可复用的检验配置，定义某类对象需要检查哪些项目、标准范围、单位、频次和判定规则。
_Avoid_: 把检验标准当成实际检验记录、在标准里保存实测值

**检验项目**:
检验标准下的一条配置项，例如余氯、pH、菌落数、中心温度、外观、异物或卫生检查项。
_Avoid_: 把检验项目当成某次检验的结果明细

**检验记录**:
一次实际发生的检验、巡检或抽检事件，指向被检对象和当时使用的检验标准。
_Avoid_: 用检验标准或检验项目冒充执行记录

**检验记录明细**:
某次检验记录下针对一个检验项目的实际值、文本结果、判定、备注和证据。
_Avoid_: 把实测值回写到检验项目配置里

**质检工作台**:
质检人员按日期、班次、生产批次、区域、任务状态组织待检事项的操作入口。它只负责聚合、筛选、分派和展示任务，不保存检验实测值、判定或不合格结论。
_Avoid_: 把质检日报、质检工作台或 `-质检` 后缀源表做成检验结果大宽表

**质检任务**:
要求质检人员在指定对象、时间或触发条件下完成某类检验/巡检的待办项，指向被检对象、检验标准、责任人、状态和完成后的检验记录。
_Avoid_: 用质检任务保存实测值、判定结果、异常结论或检验报告

**环境观察记录**:
按区域点位记录的简单环境观察值，例如温度、湿度、压差、冰柜温度或冷藏点温度。它可以按频次独立记录，也可以可选关联生产批次，但不应该为了非批次点位伪造产品批次。
_Avoid_: 把水质、虫害、卫生检查、环境微生物或多项目检验报告塞进环境观察记录

**批次过程参数记录**:
生产过程中绑定产品批次、工序、产线或生产任务的过程参数，例如料温、比重、烤炉过程参数、CCP 监控或生产过程巡检参数。
_Avoid_: 用它记录公共区域温湿度、冷库点位、虫害巡查或水质检测

**批次质量状态**:
原辅料批次因质量、过期或锁定控制产生的可用性状态。
_Avoid_: 用库存流向解释质量锁定

**批次库存流向状态**:
原辅料批次在库存流转中的状态，例如在库、已消耗、隔离、处置。
_Avoid_: lot_status、用质量状态替代库存流向

**供应商运营状态**:
供应商主数据是否可被日常业务继续使用的管理状态。
_Avoid_: 用供应商评估状态替代启用/停用

**供应商评估状态**:
供应商经采购或质量评估后的准入状态，独立于供应商是否被运营启用。
_Avoid_: supplier_status、把评估准入状态和运营启停状态合并

**外部方**:
除供应商以外，与公司发生客户、承运、废弃物收集、外包服务或其他外部协作关系的组织或个人主体。
_Avoid_: 把客户、承运商、废弃物收集方硬塞进供应商模块

**客户**:
外部方的一种类型，用于投诉、发货、召回通知和客户资料维护。
_Avoid_: 新建独立客户表和 ExternalParty 并行维护

**访客登记**:
一次外来人员进入公司或车间的访问事件，保存来访人、组织、目的、日期、陪同人、出入状态和必要快照。
_Avoid_: 用访客登记保存所有健康声明、保密承诺、外出放行或包裹检查字段

**访问/放行声明**:
围绕访客进入、员工外出、货物放行、邮件包裹检查、健康承诺、保密承诺或卫生承诺形成的声明或放行事件。
_Avoid_: 把声明字段塞进 VisitorRecord 或三级文件后系统无法追踪

**外部方评价**:
对承运商、外包服务方、废弃物收集方等非供应商外部方进行的准入、周期或风险评价。
_Avoid_: 直接复用供应商评价导致承运商进入采购供应商清单

**配料区库存**:
某个车间配料区内按原辅料批次记录的现场结存。
_Avoid_: 用仓库总库存代替配料区库存

**库存余额**:
某个物料批次、库位或暂存区在当前时点的结存数量和可用状态。
_Avoid_: 用库存流水反推当前余额、把每一次出入库动作当成余额事实

**库存流水**:
库存对象在位置、数量或状态上发生变化的一次可追溯移动记录，是仓库事实的权威记录。首发可以在仓库任务入口和产品批次详情页都提供成品入库确认入口，但两个入口必须调用同一个仓库库存流水动作；任何模块不得绕过仓库库存流水直接改库存事实。成品库存余额首发不单独建余额表，以产品批次库存流水查询或汇总为准。
_Avoid_: 把入库、出库、调拨、盘点调整、成品入库、成品出库拆成互不相干的流水事实源、只改库存余额不写流水、每个业务模块各自写库存、页面各写一套入库逻辑、为成品库存另建一套和库存流水抢写的余额表

**区域点位**:
公司内可被库存、清洁、温控、采样、虫害、设备安装、暂存或巡检引用的统一位置主数据。它可以表达车间、房间、库位、暂存区、清洁点、温控点、采样点、虫害点或设备点位。
_Avoid_: 每个场景各建一张位置表、在业务记录里手填位置文本

**WorkshopArea**:
noidear 当前承接区域点位的实现落点。业务文档使用 **区域点位** / `AreaPoint` 作为标准名，代码实现优先扩展 `WorkshopArea`，不另建平行 `AreaPoint` 表。
_Avoid_: `WorkshopArea` 和 `AreaPoint` 双表并存、同一个点位有两个 ID

**点位类型配置**:
绑定在区域点位上的专项规则，例如温控上下限、采样方法、虫害装置类型、清洁点位属性、库位属性或设备安装属性。
_Avoid_: 用 `TemperatureControlPoint`、`CleaningPoint`、`PestControlPoint` 重新保存一套区域名称和层级

**设备资产**:
生产设备、设施、计量器具、紫外灯、过滤器、空气处理部件、金属探测机等可被维护、校准、验收、使用或安装点位引用的统一资产身份。
_Avoid_: 普通设备台账、测量设备台账、设施台账、过滤器台账各自维护一套设备身份

**普通设备**:
noidear 当前用于承接生产设备、设施、维修、保养和故障的实现落点。
_Avoid_: 用普通设备直接保存计量校准读数或外检报告全文

**计量器具**:
需要检定、校准或能力确认的测量设备，例如秤、温度计、检测仪器等。
_Avoid_: 和普通设备重复建档、把校准周期写成维护计划备注

**维护计划**:
针对设备资产按日期、频次、等级或规则生成的维护、保养、润滑、过滤器更换、清灰等计划或到期任务。
_Avoid_: 把年度保养计划做成静态表、用维护记录倒推计划

**维护记录**:
某次实际维护、维修、保养、润滑、更换或清灰事件，指向设备资产和可选维护计划。
_Avoid_: 用维护计划替代执行记录、把维修申请单单独做成第二套事实源

**设备使用记录**:
仪器或设备在某次样品、实验、生产或检验中的使用流水，记录使用时间、用途、使用人和状态。
_Avoid_: 用维护记录承接日常使用流水

**校准记录**:
某次内部校验、外部检定、使用前校准或期间核查事件，指向计量器具，保存校准日期、有效期、机构、证书号、结论和证据。
_Avoid_: 把校准当成设备维护记录、把证书号当成校准事实的唯一来源

**校准读数明细**:
一次校准中的单点读数或测试块结果，例如左上、右下、中心点、标准值、实测值、误差和单点判定。
_Avoid_: 把五点校准压成一个文本结论、把读数写进附件后系统不可查询

**设备验收记录**:
设备进场、试运行、启用前的验收事件，决定设备是否可投入使用。
_Avoid_: 用设备主数据的启用日期替代验收过程和签字证据

**外部检定报告**:
第三方检定或校准机构出具的证书、报告或附件，是 **校准记录** 或 **计量器具** 的证据文件。
_Avoid_: 为每份外检报告单独建业务主表、把报告附件塞进受控文件版本

**班组**:
对某个班次生产、配料、包装或确认动作负责的现场责任组织。
_Avoid_: 只用操作人替代班组责任

**部门**:
系统管理中的组织主数据，用于承载用户归属、部门负责人、权限配置和组织初始化。当前产品口径定义为单层部门，不支持上下级层级关系。
_Avoid_: 树形部门、父子部门事实源

**部门本级权限**:
单层部门口径下授予某个部门自身的权限边界，不包含下级部门继承或扩散语义。
_Avoid_: 子部门隔离、本部门及下级、把下级部门权限退化成本部门权限

**部门负责人**:
某个部门当前的责任用户，由部门正式字段直接指定，用于组织初始化、部门治理和责任归属。负责人事实源是部门对用户的直接指派；角色只决定候选资格，不直接等于负责人身份。
_Avoid_: 仅凭角色推断负责人、从操作记录反推负责人

**系统角色**:
系统治理中用于表达管理员、部门负责人和普通用户身份边界的正式角色主数据。系统角色的正式来源是角色主数据本身，而不是用户身上的影子字符串；运行时不再保留独立影子 role 事实源。
_Avoid_: 用影子字段、临时文本或页面局部约定充当正式角色来源

**运行时角色代码**:
认证后供权限判断消费的系统角色代码，必须来自正式系统角色，不构成新的角色事实源。
_Avoid_: 把 JWT 内部字段名、导入列名或页面展示文本当成正式角色来源

**组织与权限初始化完成**:
系统第一次可用时的阶段性完成状态，表示组织与权限基础骨架已经建立完成。初始化完成后，后续负责人异常、用户未归属部门等问题属于日常治理异常，不会把系统重新打回首次初始化阶段。
_Avoid_: 把日常治理异常重新解释成首次初始化未完成、在正常初始化流程中重置完成状态

**组织治理异常**:
初始化完成之后，部门负责人缺失、负责人失效、用户未归属部门、用户挂在停用部门等组织与权限口径问题。这类问题属于日常治理，不属于首次初始化未完成。
_Avoid_: 用初始化状态描述日常治理异常

**排班**:
某个工作日期和时间班次对应到责任班组、负责人和成员的安排。
_Avoid_: 从操作时间临时猜负责人

**时间班次**:
按企业规则划分的日班、夜班等生产时间段，可跨自然日。
_Avoid_: 直接按自然日切批次责任

**班前盘点**:
上班或接班开始前，对暂存区现有原辅料批次和数量进行实物确认，目的是让当班人员知道自己接手了哪些物料、数量是否对得上。它同时承接上一班的 **班后盘点**，由接班人确认“上一班交出的”和“我现场看到的”是否一致；不需要再额外做一张独立的接班盘点。
_Avoid_: 用月度盘点替代班次接手确认、把班前盘点做成复杂开工审批、另建独立接班盘点

**班后盘点**:
下班或本班结束前，对暂存区剩余原辅料批次和数量进行实物确认，作为交给下一班或结束本班责任的依据。
_Avoid_: 不盘点就默认交班

**盘点差异**:
暂存区账面库存和实物库存不一致的事实。首发先作为盘点记录中的差异状态和备注处理，不默认上升为正式不合格或整改；确实影响生产或追溯时，再由负责人转入异常处理。
_Avoid_: 默认作为普通盘盈盘亏跳过、所有轻微差异都强制走复杂整改流程

### 追溯与质量闭环

**公司 ID**:
系统中用于隔离不同公司数据的租户边界标识；同一业务链只能有一套公司边界。
_Avoid_: 在业务服务中硬编码 company_id、把操作人 ID 当成公司边界、同一主链并存两套公司 ID

**追溯主链**:
从原辅料批次经投料关系到产品批次、发货和客户影响范围的正式关系链。
_Avoid_: 用备注、附件或自由文本当正式追溯关系

**正向追溯**:
从上游原辅料批次或产品批次向下游产品、发货和客户影响范围展开的查询。
_Avoid_: 只查当前库存

**反向追溯**:
从发货、投诉、召回或产品批次向上游投料、原辅料批次和供应商展开的查询。
_Avoid_: 只按客户口述批号搜索

**追溯快照**:
某一次正向或反向追溯查询结果的不可变结构化快照，保存根对象、查询范围、上下游结果、摘要和导出文件引用。
_Avoid_: 把追溯快照当成演练活动、每次证据导出都重新实时查链路

**物料平衡**:
围绕原辅料批次或产品批次核对入库、移动、投料、产出、损耗、留样和报废的数量闭环。
_Avoid_: 只看单张投料表判断是否平衡

**留样**:
从产品、原料、包材或样品中按制度留存的实物样品，承载样品来源、批次、数量、保存条件、留样日期、到期日期、状态和处置。
_Avoid_: 把留样当成普通库存、把留样检查和保质期实验结果直接塞进留样主表

**留样检查**:
针对某个留样样品在保存期间或到期时进行的检查、复检、处置确认或检验事件。
_Avoid_: 用留样主表字段替代一次检查记录、把检验实测值写进留样档案

**保质期实验**:
围绕某个产品、配方、批次或留样样品设计的周期性稳定性/保质期验证活动，包含保存条件、实验周期、节点计划、节点检验和最终结论。
_Avoid_: 把每个节点做成互不相干的检验记录、用单次留样检查替代整个实验

**保质期实验节点**:
保质期实验中的一次计划检查节点，例如 0 天、1 个月、3 个月、6 个月或到期后复核，可触发检验记录或留样检查。
_Avoid_: 在保质期实验主表里横向展开一堆月份字段

**追溯演练**:
用真实或模拟问题批次验证系统能否在规定时间内完成正反向追溯和证据汇总的治理活动，必须引用一个或多个追溯快照。
_Avoid_: 普通查询记录、用追溯快照替代演练主对象

**召回**:
针对已发出产品确定影响范围、通知对象、处置动作和关闭证据的真实质量事件，必须引用冻结后的追溯快照确定影响范围。
_Avoid_: 普通投诉备注、每次打开召回都实时重新计算影响范围

**不合格**:
产品、物料、过程、设备、检验明细、清洁明细、投诉或文件执行中被判定不满足要求的质量事件本体。
_Avoid_: 只写在备注里的问题、把整改措施或验证结论塞进不合格本体

**整改措施**:
针对不合格、投诉、审核发现或变更问题制定的原因分析、纠正措施、预防措施、责任人和期限。
_Avoid_: 把整改措施当成问题来源、只改不合格状态不建措施

**验证记录**:
对整改措施是否执行完成、是否有效、是否可以关闭的验证事实和证据。
_Avoid_: 用口头确认或直接关闭状态替代验证记录

**整改闭环**:
围绕不合格、投诉、审核发现或变更问题形成的来源定位、原因分析、措施、验证和关闭过程。
_Avoid_: 只创建待办不验证效果、把不合格/整改/验证合成一个宽表

**清洁方案模板**:
某类区域的默认清洁/消毒方案配置，例如干区、油脂区、鸡蛋处理区、厕所或后勤区域。它用于初始化具体区域的清洁方案，不保存任何执行事实。
_Avoid_: 把称油间、厕所一号、鸡蛋房等具体区域硬编码成不同表单或页面

**清洁方案**:
某个具体区域当前生效的清洁/消毒方案，类似“清洁版配方”，由区域、版本、频次和清洁项目组成。
_Avoid_: 每次清洁时临时手填本区域有哪些地面、墙壁、天花板、门窗或工具要清洁

**清洁方案明细**:
清洁方案中的项目行，定义要清洁或消毒的对象、点位、方法、频次、是否消毒、消毒剂、目标浓度、正常范围、是否必做和是否复核。
_Avoid_: 把目标浓度、清洁方法和必做项写成执行记录里的自由文本

**清洁记录**:
某一次清洁执行的主记录，指向具体区域、清洁方案、执行任务、执行人和执行时间。
_Avoid_: 用一张“日常清洁记录表（某区域）”宽表同时保存方案配置和执行结果

**清洁记录明细**:
某次清洁执行中每个清洁项目的实际完成结果，例如完成/未完成、结果、备注、复核、照片或证据引用。
_Avoid_: 把它当成清洁项目配置表、把目标浓度和规则写回执行明细

**消毒液浓度检查**:
针对消毒液、余氯、脚底清洁池、鸡蛋房消毒池或洗手消毒水等浓度的专项检查记录，按区域、点位、容器或消毒剂保存目标值、实测值、单位、判定和异常。
_Avoid_: 把浓度检查塞进普通清洁记录明细、让每张清洁表各自维护一套浓度字段

### 文控与记录表单

**体系文件中心**:
受控文件的统一入口，承载文件库、台账、版本、审批、引用、复审、到期和生命周期。
_Avoid_: 体系文件库和文档台账并列成两个主入口

**受控文件**:
纳入文控生命周期的管理手册、程序文件、作业指导书、公司文件、外来文件或受控附件。
_Avoid_: 普通网盘文件、业务证据附件、某次导出的记录 PDF

**文件修订**:
对已发布受控文件发起新版本草稿、审批并发布的换版过程。
_Avoid_: 原地修改已发布文件、用文件修订承接业务对象变更或导出模板版本

**导出模板**:
用于生成记录 PDF、审核证据包、追溯报告或客户报告的版式、字段绑定、审批信息区和模板版本。
_Avoid_: 把导出模板当成受控文件本体、把模板版本和业务数据快照混在一起

**证据文件**:
绑定到某个业务对象或业务对象明细上的附件、照片、报告或生成文件。
_Avoid_: 把所有业务附件都塞进受控文件库、用证据文件替代业务对象状态

**证据导出**:
一次实际导出的审核证据、记录 PDF、追溯包或报告快照，保存导出模板版本、业务数据快照、导出人、导出时间和输出文件。
_Avoid_: 覆盖历史导出、把导出结果当成受控文件版本

**证据数据快照**:
证据导出中冻结的结构化业务事实。它必须同时保存对象 ID 和当时可读事实，例如名称、编号、批号、状态、数量、单位、检验结论、审批意见、异常状态和关键时间；不能只保存 ID 后续再实时联查。
_Avoid_: ID-only 数据快照、打开旧证据包时按当前主数据名称或当前检验标准重新渲染

**一键证据导出**:
用户从产品批次等业务页面直接触发的证据导出动作。首发阶段点击导出时系统自动先冻结新的追溯快照，再生成证据导出；高级追溯页面可以从已有追溯快照再次导出。
_Avoid_: 要求普通用户先手工生成追溯快照、导出文件打开时重新实时查询链路

**证据包生成与重新下载**:
同一产品批次可以多次生成证据包；每次“生成新证据包”都是新的追溯快照和新的证据导出。历史证据包只能重新下载或查看原文件，不能覆盖、刷新或复用为新的导出。
_Avoid_: upsert 覆盖旧 EvidenceExport、把“重新下载旧文件”和“重新生成证据包”做成同一个动作

**正式证据包**:
可作为审核证据使用的首发主链证据包。它只能在 `ProductionBatch.status = completed`、主链必需事实完整且没有未关闭不合格/整改时生成；`released_at / released_by` 如存在必须冻结进证据数据快照，但首发不把它们作为硬门槛。
_Avoid_: 把未完成批次、未关闭异常或缺主链事实的当前查询结果当成正式审核证据

**当前快照预览**:
用于现场排查或管理查看的当前追溯状态预览。它可以展示 incomplete 状态和缺口原因，但不能作为正式证据包，也不能生成正式 EvidenceExport。
_Avoid_: 用预览 PDF 代替正式证据包、把 incomplete 快照拿去审核

**审批/审核快照**:
某次证据导出或业务关闭时冻结的审批事实摘要，包含提交人、审核/审批人、审批时间、审批结论和当时的审批意见。
_Avoid_: 手写签名图片、CA 电子签章、复杂签字区排版

**首发主链证据包**:
首发闭环切片中用于证明主链已经跑通的一组导出证据，覆盖来料、检验、批次、库存、生产计划、投料、产品批次、异常整改、追溯快照和关键附件。它证明对象关系、状态流、检验结论、异常整改和追溯范围完整，不负责还原全部源表纸面版式。
_Avoid_: 把首发证据包理解成 282 张源表模板全部复刻

**首发证据包根对象**:
首发主链证据包的导出入口对象。首发阶段唯一根对象是 **产品批次**；来料、检验、原辅料批次、库存、投料、异常、整改、审批和证据文件只能作为该产品批次证据链里的节点出现。
_Avoid_: 首发支持从供应商、来料单、原辅料批次、检验记录、异常单等任意对象直接导出证据包

**下游影响范围**:
从产品批次向发货单、客户、投诉、召回通知或召回处置展开的影响对象集合。首发主链证据包只预留这个区块；已有可靠发货或召回关联时可以展示，缺失时不阻塞首发主链验收。
_Avoid_: 为了首发证据包强行补完整销售/发货模块、把发货/客户链不完整视为主链失败

**证据摘要文件**:
用户下载查看的首发主链证据包摘要文件，通常是简单 PDF 或 HTML 渲染出的 PDF，展示主链对象、状态、检验结论、异常整改、审批/审核快照、追溯快照和附件索引。
_Avoid_: 把 Excel 当首发证据包主格式、把所有原始附件打包成 zip、用摘要文件替代系统内 JSON 快照

**记录表单索引**:
04 记录表单的主索引（已退役）。动态表单平台（RecordTemplate / Record / Task / TaskRecord 等 9 个模型）已于 2026-05-24 完全退役，见 `docs/adr/0001-retire-dynamic-form-platform.md`。源表单现由各独立业务 `*Record` 模型直接承接，无需中间索引层。
_Avoid_: 把 04 当普通文件夹或所有表单的统一填写页

**动态表单**（已退役）:
原用于承接不适合独立业务建模的记录模板和填写记录。动态表单平台已完全退役，见 `docs/adr/0001-retire-dynamic-form-platform.md`。
_Avoid_: 在已退役平台上新建任何记录模板或填写流程

**业务模块承接**:
源记录表单由研发、采购、仓储、生产、品质、设备等模块中的正式业务 `*Record` 对象直接承载，不再经由动态表单层中转。
_Avoid_: 已有业务对象还重复建动态表单事实源

**工作台会话**:
现场用户在一个页面连续处理多个业务动作时产生的操作上下文，只保存班次、区域、人员、任务、状态和真实业务对象引用。
_Avoid_: 把鸡蛋房日报、洗衣房工作记录或班后检查整表做成事实源

**落地健康状态**:
记录表单索引中用于判断源表单是否已正确关联入口、模板、业务对象和执行证据的状态。
_Avoid_: 只显示有无路由

**引用健康状态**:
体系文件 Markdown 引用是否正常、悬空、无效、冲突或指向旧版本的判断。
_Avoid_: 只展示引用数量

**Excel 导入导出**:
系统中用于统计导出、记录导出和业务报表下载的电子表格交换能力，由后端负责生成。应用内批量导入不作为产品功能保留；初始化或大批量数据灌入走开发/运维脚本，不放在业务 Web/H5 页面里。
_Avoid_: 前端本地解析 Excel、浏览器端 xlsx 依赖、页面各自拼装 Excel 文件、在产品功能里维护通用批量导入中心

**Web/H5 主系统**:
用户当前通过浏览器访问的主业务系统，承载工作台、文控、记录、仓储、质量、设备、追溯和系统治理等页面。
_Avoid_: 把未设计的移动端草稿工程当成主系统 H5 页面

**移动端交付面**:
独立设计给手机 App、小程序、离线采集或现场拍照同步使用的移动产品面。
_Avoid_: 响应式 CSS、浏览器页面的手机适配、Web/H5 主系统

**应用内备份管理**:
在业务 Web/H5 主系统内手动触发数据库或对象存储备份、查看备份历史和删除备份记录的运维页面能力。备份属于部署运维基线，不作为当前业务系统功能保留。
_Avoid_: 在业务应用内用 docker exec 触发备份、把备份历史当业务记录

**应用内健康管理**:
在业务 Web/H5 主系统或业务 API 中查看数据库、缓存、对象存储、磁盘和系统信息的运维能力。它不作为当前业务系统功能保留。
_Avoid_: 把依赖服务健康详情、磁盘信息、系统信息做成业务功能

**运行存活探针**:
供 Docker、Kubernetes、反向代理或部署平台判断服务进程是否存活的最小探针。
_Avoid_: 依赖健康详情、业务健康度、运维大屏

**通用回收站**:
跨文档、记录模板、记录等多个业务对象提供统一查询、恢复、永久删除和自动清理的横向恢复入口。当前不作为业务系统功能保留。
_Avoid_: 用一个通用恢复入口替代各业务对象自己的归档、作废、撤销或删除规则

**通用批量导出中心**:
跨用户、文档、任务、记录、偏差等多个业务对象提供统一导出入口的后台工具页和通用导出 API。当前不作为业务系统功能保留。
_Avoid_: 用一个 `/export/*` 中心替代业务页面里的具体导出动作

**应用内批量导入**:
在业务 Web/H5 系统中提供通用 Excel 模板下载、预览和导入页面。当前不作为产品功能保留；必要数据由开发/运维直接灌入。
_Avoid_: 为一次性初始化需求维护长期 UI、后端导入 API、CLI 替代入口和模板预览流程

**业务模块导出**:
由具体业务模块拥有的导出动作，服务该模块页面里的真实业务场景，例如偏差报告、统计、审计日志、追溯、记录 PDF 或批次 PDF。
_Avoid_: 让业务导出继续依赖通用 `/export/*` 中心、导出内部 ID 替代业务可读名称

**记录填写结果导出**（已退役）:
动态表单平台退役后，此能力随 Record/RecordTemplate 模型同步退役，见 `docs/adr/0001-retire-dynamic-form-platform.md`。动态表单平台退役后，各独立业务 `*Record` 模型负责自身导出，不再依赖通用 Record 导出路径。
_Avoid_: 继续引用 `/export/task-records` 或 `Record + RecordTemplate` 作为导出事实源

### 任务、审批与工作流

**正式审批**:
对业务对象作出同意、驳回、转交或撤销等决策的审批事实。
_Avoid_: 用普通待办或工作流任务替代审批事实

**历史审批记录**:
旧审批表中已经产生的文档审批证据，仅用于历史兼容、审计查询和签名显示。
_Avoid_: 给新文档继续写旧审批记录、把历史兼容路径当成新审批入口

**待办收件箱**:
用户需要处理事项的统一入口投影。
_Avoid_: 把待办当成审批、记录填写或业务执行事实源

**记录填写任务**（已退役）:
原用于要求用户在指定时间或条件下填写某张记录表单。`record-task` 模块已随动态表单平台退役，见 `docs/adr/0001-retire-dynamic-form-platform.md`。
_Avoid_: 在已退役模块上新建记录填写任务

**周期任务生成器**（已退役）:
原用于按频率、班次、日期或规则生成记录填写任务。`scheduled-task` 模块已随动态表单平台退役，见 `docs/adr/0001-retire-dynamic-form-platform.md`。
_Avoid_: 在已退役模块上新建周期调度规则

**工作流编排**:
把多个任务、审批或状态节点串成流程的编排能力。
_Avoid_: 用工作流编排另造第二套审批事实源

## Relationships

- 一个 **产品档案** 可以有多个 **配方** 版本，但新生产只能使用可用产品和可用配方。
- **产品画像** 汇总 **产品档案**、**配方**、**产品标签画像**、**产品过敏原画像**、**产品风险区判定**、**产品规格书** 和 **产品验证记录**，但不成为这些对象的事实源。
- **物料过敏原画像** 是过敏原的源头事实；**产品过敏原画像** 由 **配方明细** 和 **物料过敏原画像** 自动生成。
- **产品标签画像** 由 **产品档案**、**配方**、**产品过敏原画像**、**企业资质画像** 和法规字段生成；标签变化属于 **变更事件**，不能绕过配方和过敏原事实源手填。
- **产品规格书** 是导出快照；每次导出应绑定 **导出模板**、数据快照和 **证据文件**，不能反向维护产品、配方或标签事实。
- **产品风险区判定** 和 **产品验证记录** 属于产品开发/变更阶段事实；它们可以引用检验、变更和证据，但不替代检验记录或三级文件。
- 一个 **配方** 包含一条或多条 **配方明细**。
- 一条 **配方明细** 可以指定一个 **配料区库存** 所属的配料区域。
- **产品归档** 会让相关 **配方** 和 **工序** 退出正常业务，但不改写历史 **产品批次**。
- **生产计划** 包含一条或多条 **生产计划明细**；发布后派生 **现场执行任务**，但不提前生成 **产品批次**。
- **生产运行** 可以作为现场执行过程的上下文，但不是首发主链必经节点，也不能替代 **生产计划**、**现场执行任务** 或 **产品批次**。
- **产品批次** 在包装机喷码、包装确认或成品入库确认时创建或确认，并回链到对应 **生产计划明细**。
- **产品批次确认** 只表示“这个批号开始了”，不得写成品入库 **库存流水**，也不得把产品批次改成 `completed`。
- **包装确认** 由包装岗位提交，记录包装、喷码、数量等生产事实，可以创建或确认 **产品批次**，但只形成待入库事实，不生成成品入库 **库存流水**。
- **成品入库确认** 由仓库或成品暂存区接收人完成，才生成成品入库 **库存流水**（`object_type = production_batch`、`movement_type = production_in`），因为库存事实从仓库或成品暂存区接收开始；同一个 **产品批次** 关闭前可以有多条成品入库流水。
- **成品入库确认** 复用现有 **仓库模块**，只扩展产品批次库存流水能力；不得新建平行的成品入库模块、包装入库模块或第二套仓库库存流水。
- **成品入库确认** 可以从仓库任务入口或产品批次详情页触发，但两个入口必须共用同一个仓库库存流水动作，不得各自实现一套入库逻辑。
- 所有库存增减都必须落到 **库存流水**；首发不为成品库存新增独立余额表，成品库存以 `production_batch` 相关库存流水查询或汇总为准。
- **成品入库确认** 只能选择已有 **产品批次**，不得修改产品批次号；首发不新增现场实物批号扫描、录入或自动比对功能。
- 当前不新增独立 **包装确认** 或 **包装产出记录** 对象，也不校验成品入库数量小于等于包装累计数量；成品入库数量以仓库或成品暂存区接收确认为准。
- **产品批次** 只有经过 **批次关闭确认** 才能进入 `completed`；只要该批号还可能继续产生包装或成品入库事实，就必须保持 `in_progress`。
- 连续生产时，切换到下一批号是上一产品批次关闭确认的强触发；非连续生产时，没有下一批号也可以手动关闭当前产品批次。
- **批次关闭确认** 的前置条件是该产品批次已经存在至少一条成品入库 **库存流水**（`object_type = production_batch`、`movement_type = production_in`）；没有成品入库库存流水时不得关闭。
- **批次关闭确认** 不新增关闭人、关闭时间、关闭原因、实际产量快照或入库量快照；成品入库量和库存变化由 **库存流水** 证明。
- **产品批次** 进入 `completed` 后，不得继续追加普通成品入库 **库存流水**（`movement_type = production_in`）。漏入库、补录或更正只能走 **库存调整**（`movement_type = adjustment`），并在流水里指向该产品批次。
- 一个 **产品批次** 可以归集一个或多个 **配料投入池**。
- 一个 **配料投入池** 可以通过执行归集关系关联一个或多个 **产品批次**。
- 一个 **配料执行** 包含一条或多条 **原辅料批次** 投入明细。
- **投料记录** 是产品批次与原辅料批次的追溯桥接结果，由 **配料执行** 明细确认后自动生成；现场人员只填 **配料执行** 和明细，不直接维护投料追溯桥。
- **执行归集关系** 只说明哪些 **配料执行** 或 **配料投入池** 归到哪个 **产品批次**；它不保存逐物料批次用量，逐物料批次用量由 **投料记录** 承接。
- **配料执行** 扣减 **配料区库存**，不是直接扣减仓库总库存。
- **客户** 是 **外部方** 的一种类型；客户投诉、发货和召回通知引用外部方客户，不新建平行客户主数据。
- 承运商、废弃物收集方和外包服务方也属于 **外部方**，但不属于 **供应商**，不能直接进入供应商准入/绩效评价。
- **外部方评价** 只评价非供应商外部方；供应商仍走 **供应商评估状态** 和供应商评价链路。
- **访客登记** 是访问事件；**访问/放行声明** 是声明或放行事件。一次访客登记可以关联一条或多条声明，但不能把所有声明字段硬塞进访客登记宽表。
- **库存余额** 表达当前还剩多少、在哪里、状态是什么；**库存流水** 表达为什么变化、何时变化、由谁触发。
- **库存流水** 是长期统一库存移动事实源，应覆盖原辅料入库、领料、调拨、退料、报废、盘点调整、成品入库和成品出库；任何改变 **库存余额** 的动作都必须同步写入 **库存流水**。
- **区域点位** 是库存位置、清洁点、温控点、采样点、虫害点、设备安装点和暂存区的统一身份；专项对象只保存规则或记录，不重新定义物理位置。
- `WorkshopArea` 是 **区域点位** 的当前代码落点；开发应扩展它的类型、父级、公司、风险和规则字段，而不是新建平行 `AreaPoint` 表。
- 温控、采样、虫害、清洁、库位和设备安装等差异通过 **点位类型配置** 表达；它们引用同一个 **区域点位**。
- **设备资产** 是普通设备、设施和计量器具的业务统称；noidear 当前落点可以继续分为 **普通设备** 和 **计量器具**，但不得形成两套设备身份。
- **普通设备** 承接维护、维修、保养、故障、设施维护、过滤器更换、金属探测机维护等事件；**计量器具** 承接校准、检定、期间核查和能力属性。
- **维护计划** 生成或组织到期维护动作；**维护记录** 承接实际执行结果。所有年度保养、润滑、过滤器更换和设备设施维护都应进入这一组边界。
- **校准记录** 承接校准主事实；**校准读数明细** 承接多点读数和单点判定；**外部检定报告** 作为 **证据文件** 绑定到计量器具或校准记录。
- **设备验收记录** 决定设备能否启用；启用日期或状态不能替代验收事件。
- **来料登记** 先形成来料明细；只有通过 **来料检验门禁** 后，才生成可用 **原辅料批次** 和入库 **库存流水**。
- **检验标准** 包含一条或多条 **检验项目**；它们是配置，不保存现场实测值。
- **检验记录** 引用当时生效的 **检验标准** 或标准快照，并指向被检对象。
- **检验记录明细** 引用 **检验项目** 或项目快照，保存实际值、文本结果和判定。
- **质检工作台** 只组织和展示 **质检任务**；它不是检验事实源。
- **质检任务** 指向被检对象和适用 **检验标准**，完成时创建或关联 **检验记录**；它不能保存检验结果本身。
- 一个 **检验记录** 可以由 **质检任务** 触发，也可以由来料、生产、环境、水质、卫生、虫害等业务入口触发；无论入口是什么，实际值和判定都落到 **检验记录明细**。
- 不合格的 **检验记录明细** 可以生成 **不合格**，并通过来源字段精确回链到明细。
- **环境观察记录** 只承接简单点位观察，例如温湿度、压差、冰柜温度；被观察对象是 **区域点位**，产品批次只能作为可选上下文。
- **批次过程参数记录** 必须绑定生产批次、工序或生产任务上下文；它承接生产过程参数，不承接独立环境点位。
- 水质、虫害、卫生检查、车辆卫生、过敏原测试、环境微生物检验等需要模板、项目、判定或报告的场景走 **检验记录** / **检验记录明细**，不走 **环境观察记录** 或 **批次过程参数记录**。
- **清洁方案模板** 用于复制生成具体区域的 **清洁方案**；模板本身不保存执行事实。
- 一个 **清洁方案** 属于一个 **区域点位**，包含一条或多条 **清洁方案明细**，并通过版本和生效状态控制现场可用性。
- **清洁记录** 引用当时生效的 **清洁方案** 和具体区域；**清洁记录明细** 引用对应 **清洁方案明细** 或项目快照，只保存实际执行结果。
- **消毒液浓度检查** 是独立专项记录，可以被 **清洁记录明细** 引用作为证据，但不并入普通清洁记录明细字段。
- 不合格的 **清洁记录明细** 或 **消毒液浓度检查** 可以生成 **不合格**，并通过来源字段精确回链到明细或检查记录。
- **不合格** 只保存问题事件本体；来源由 `source_type/source_id/source_item_id` 精确指向检验明细、清洁明细、环境记录、投诉、虫害巡查等来源。
- **整改措施** 由 **不合格**、投诉、内审发现、追溯演练或召回触发；它保存原因、措施、责任人、期限和执行状态。
- **验证记录** 关闭 **整改措施**；异常关闭前必须能看到对应措施和验证结果。
- 夜班跨零点时，同一班次可以产生两个 **产品批次**，例如 20260429 和 20260430。
- 一个 **排班** 把 **时间班次** 绑定到一个 **班组**。
- **班后盘点** 由上一班完成，**班前盘点** 由下一班完成；下一班的班前盘点就是对上一班班后盘点的承接确认，不再单独建 **接班盘点**。
- **盘点差异** 不默认阻断整个生产，但对应物料批次被继续使用前，应至少有差异备注或负责人确认；确实影响生产或追溯时再转入异常处理。
- **追溯主链** 以 **原辅料批次**、**配料执行**、**配料投入池** 和 **产品批次** 的关系为核心。
- **正向追溯** 从上游批次展开到下游产品和客户影响范围。
- **反向追溯** 从下游批次、发货、投诉或召回展开到上游原辅料批次和供应商。
- **留样** 必须引用产品、原辅料批次、产品批次或样品来源；留样数量、保存条件和状态属于留样主对象。
- **留样检查** 是事件，通常引用 **检验记录** / **检验记录明细** 保存实测值和判定；留样主对象只回写当前状态和最近检查摘要。
- **保质期实验** 是计划化验证活动，包含多个 **保质期实验节点**；每个节点可以触发 **留样检查** 或 **检验记录**，最终结论回到实验主对象和产品画像。
- **追溯快照** 冻结一次正向或反向追溯查询结果；它是证据基础，不是演练或召回主事件。
- **追溯演练** 引用一个或多个 **追溯快照**，记录演练日期、用时、参与人、结论、问题和整改。
- **召回** 引用 **追溯快照** 确定影响范围，再记录通知、处置、关闭和召回内证据。
- **物料平衡** 使用库存移动、盘点、投料、产出、留样、损耗和报废共同解释数量闭环。
- **整改闭环** 可以来源于 **不合格**、投诉、内审发现、追溯演练或召回，但处理链路必须按 **不合格** -> **整改措施** -> **验证记录** 分层。
- **体系文件中心** 管 **受控文件** 的版本生命周期；**记录表单索引** 管 04 记录表单的落地关系。
- **导出模板** 管版式和字段绑定；**证据导出** 管某一次导出行为和当时的数据快照；**证据文件** 管导出后或业务上传后的文件绑定。
- **证据数据快照** 必须冻结对象 ID 和当时可读事实；供应商、物料、批次、检验、审批、异常等后续改名或换版，不得改变旧证据包呈现。
- **一键证据导出** 从业务页面触发时必须先冻结新的 **追溯快照**，再生成 **证据导出**；高级追溯页面可以从已有快照再次导出。
- **证据包生成与重新下载** 是两个动作：生成新证据包会创建新的 **追溯快照** 和 **证据导出**；重新下载历史证据包只读取原导出文件和原快照。
- **正式证据包** 需要 `ProductionBatch.status = completed`、主链必需事实完整且没有未关闭不合格/整改；`released_at / released_by` 如存在要冻结进快照，但首发不强制要求；否则只能生成 **当前快照预览**，并标记 incomplete 和缺口原因。
- **审批/审核快照** 只冻结谁提交、谁审核/审批、审批时间、审批结论和当时审批意见；它不承接手写签名图片、CA 电子签章或复杂签字区排版。
- **首发主链证据包** 是 **证据导出** 的首发最小形态，只要求导出主链对象、状态、检验、异常、整改、审批/审核快照、追溯快照和关键附件索引；完整源表版式导出属于后续 **导出模板** 治理。
- **首发证据包根对象** 只允许是 **产品批次**；证据包从产品批次向上展开来料、检验、原辅料批次、库存、计划和投料，向下展开异常、整改、追溯快照和关键证据。
- **下游影响范围** 在首发主链证据包中是可选区块；已有 `DeliveryNote`、召回影响范围或客户关联时展示，缺失时不得让首发证据包失败。
- **证据导出** 的系统内权威数据是 JSON 数据快照；**证据摘要文件** 只是用户下载查看的呈现结果。
- **证据摘要文件** 首发可用简单 PDF 或 HTML/PDF；附件只列文件名、链接、hash、上传人和上传时间，不把原始附件打包进 zip；Excel 不作为首发证据包主格式。
- **Document / DocumentVersion** 承接受控文件和文件版本；它们不能替代 **证据文件**、**证据导出** 或 **导出模板** 的业务边界。
- 召回内的 `ProductRecallEvidence` 只承接召回事件内证据；通用审核包、追溯包和记录 PDF 仍按 **证据导出** / **证据文件** 处理。
- 一个 **记录表单索引** 条目由 **业务模块**（各独立 `*Record` 模型）直接承接；**动态表单平台已于 2026-05-24 退役**，见 `docs/adr/0001-retire-dynamic-form-platform.md`。
- **工作台会话** 可以把多个现场动作组合成一页操作，但库存、环境、清洁、浓度、计量、检验和异常等事实必须分别落到对应业务对象。
- **引用健康状态** 属于受控文件关系；**落地健康状态** 属于记录表单落地关系。
- **Excel 导出** 可以服务记录、文控、培训和系统治理等模块，但文件生成事实统一归后端服务；**应用内批量导入** 不保留为产品功能。
- **Web/H5 主系统** 可以有移动端响应式显示；这不等于启用了独立 **移动端交付面**。
- **MinIO** 承载业务文件存储；**应用内备份管理** 不承载业务文件事实，剔除它不等于删除文件上传、预览或归档能力。
- **应用内健康管理** 剔除后仍保留 **运行存活探针**；这不影响文控 **引用健康状态**、访客健康状态或员工健康记录等业务语义。
- **通用回收站** 剔除后，业务对象仍可保留 `deletedAt`、归档、作废或撤销等自身状态；是否允许恢复由各业务模块规则决定。
- **通用批量导出中心** 剔除后，**业务模块导出** 仍可保留；偏差报告、统计、审计日志、追溯、记录 PDF、批次 PDF、培训统计等应由各自业务模块承接。
- **记录填写结果导出**（已退役）：动态表单平台退役后，此能力随 Record/RecordTemplate 模型同步退役，见 `docs/adr/0001-retire-dynamic-form-platform.md`。各独立业务 `*Record` 模型负责自身导出，不再依赖通用 Record 导出路径。
- **正式审批** 承载审批决策事实；**待办收件箱** 只投影用户需要处理的事项。
- 新文档的 **正式审批** 由统一审批实例承载；**历史审批记录** 只服务旧文档兼容和证据展示。
- **记录填写任务**（已退役）/**周期任务生成器**（已退役）：随 `record-task` 和 `scheduled-task` 模块退役，见 `docs/adr/0001-retire-dynamic-form-platform.md`。
- **工作流编排** 可以串联任务和审批，但不得另造第二套审批事实源。
- **部门本级权限** 只作用于一个 **部门** 本身；单层部门模型下不存在子部门继承权限。
- **组织与权限初始化完成** 后，**组织治理异常** 只能作为治理问题暴露，不能让系统回退到首次初始化阶段。

## Example dialogue

> **Dev:** "夜班 23:30 配了一次料，00:00 后换了批次，要建两个生产批次还是两个成品批次？"
> **Domain expert:** "业务上只叫产品批次。23:30 的配料执行形成投入池，后面可以同时归集到 20260429 和 20260430 两个产品批次。"

> **Dev:** "发布生产计划时要不要先生成产品批次？"
> **Domain expert:** "不要。生产计划只生成计划明细和现场执行任务，产品批次等包装机喷码、包装确认或成品入库确认时才创建或确认。"

> **Dev:** "确认产品批次号出现时，能不能顺便当成入库和完成？"
> **Domain expert:** "不能。确认产品批次只表示这个批号开始了；仓库收到成品要走成品入库确认；这个批号结束要走批次关闭确认。开始、入库、结束是三件事。"

> **Dev:** "那 ProductionRun 能不能先当生产计划或者生产任务用？"
> **Domain expert:** "不能。ProductionRun 只能表达现场某段运行上下文；计划用 ProductionPlan，待办用 ProductionTask，追溯批号用 ProductionBatch。"

> **Dev:** "BatchMaterialUsage 要不要做成人工补录页面？"
> **Domain expert:** "不要。现场只填 MixingExecutionLine；确认投料后系统自动生成 BatchMaterialUsage，作为追溯索引。"

> **Dev:** "BatchMixingAggregation 能不能替代 BatchMaterialUsage？"
> **Domain expert:** "不能。BatchMixingAggregation 只说明哪些配料/投料执行归到哪个产品批次；物料批次用了多少，仍由 BatchMaterialUsage 承接。"

> **Dev:** "库存只改当前数量，不写 InventoryMovement 可以吗？"
> **Domain expert:** "不行。当前数量是库存余额，出入库、领料、退料、报废、盘点调整这些变化必须写库存流水，否则以后追溯不到为什么变。"

> **Dev:** "清洁点、温控点、采样点、虫害点是不是都各自建一张位置表？"
> **Domain expert:** "不要。它们的物理身份统一是 AreaPoint，当前代码落到 WorkshopArea；TemperatureControlPoint、PestControlPoint 这类对象只保存专项规则或记录，不再保存一套区域层级。"

> **Dev:** "物料批次里的库位、设备安装位置、清洁区域可以继续用文本吗？"
> **Domain expert:** "新链路不要。位置要关联 AreaPoint/WorkshopArea，文本只能作为历史显示或导入线索，不能当权威位置事实。"

> **Dev:** "设备台账、车间测量设备台账、化验室仪器检定表是不是分别做页面？"
> **Domain expert:** "不要分别造身份。业务上统一叫设备资产；当前代码可继续用 Equipment 和 MeasuringEquipment 两个落点，但同一个资产不能在两边各有一个互不相干的 ID。"

> **Dev:** "过滤器更换、润滑、金属探测机保养、设施维修要不要各做一张记录表？"
> **Domain expert:** "不要。它们都是维护计划和维护记录的不同类型或明细；只在模板、项目或类别上区分。"

> **Dev:** "外检报告要不要做成一张外检报告表？"
> **Domain expert:** "不要。外检报告是校准记录或计量器具的证据文件；校准事实在 CalibrationRecord，读数在 CalibrationPointReading，附件在 EvidenceFile。"

> **Dev:** "客户要不要建 Customer 表？"
> **Domain expert:** "不要和 ExternalParty 并行。客户是 ExternalParty 的一种类型，投诉、发货和召回通知都引用 ExternalParty(type=customer)。"

> **Dev:** "承运商、废弃物收集方和外包服务方是不是供应商？"
> **Domain expert:** "不是采购供应商。它们是 ExternalParty，评价走 ExternalPartyEvaluation，不要塞进 SupplierEvaluation。"

> **Dev:** "访客健康声明、保密声明、员工外出放行、包裹检查能不能都加到 VisitorRecord？"
> **Domain expert:** "不要。VisitorRecord 只是一条来访事件；声明和放行统一走 AccessDeclaration，必要时关联访客、员工、外部方或货物/包裹。"

> **Dev:** "留样、留样检查、保质期实验是不是一个页面一张表就够？"
> **Domain expert:** "页面可以合在一个留样模块，但事实要分层。RetainedSample 管样品和状态，RetainedSampleInspection 管某次检查，ShelfLifeStudy 管整个实验和节点计划，实测值仍落到 InspectionRecord/Item。"

> **Dev:** "保质期实验 1 个月、3 个月、6 个月要不要在主表里加一堆字段？"
> **Domain expert:** "不要横向加字段。它们是实验节点或节点任务，每个节点触发检查或检验记录，最后回写实验结论。"

> **Dev:** "历史产品要不要重新走研发流程，还是建一张历史产品表？"
> **Domain expert:** "不用。历史产品建档和新产品研发最终都落到产品档案、配方和配方明细，只是来源不同。"

> **Dev:** "产品标签信息记录和产品规格书能不能各做一张可编辑表？"
> **Domain expert:** "不要。标签画像和规格书都是由产品、配方、过敏原、企业资质和模板生成的派生结果。能编辑的是源头事实；导出的规格书只保存快照和证据。"

> **Dev:** "产品过敏原清单是不是让品质或研发手填？"
> **Domain expert:** "不要。物料过敏原画像才是源头，产品过敏原画像由生效配方自动汇总。人工只能维护物料/供应商物料过敏原，不能直接改产品结论。"

> **Dev:** "产品风险区判定和产品验证记录要不要每批生产都填？"
> **Domain expert:** "不要。它们是产品开发或变更阶段的判定和验证事件，批次生产只引用当前生效结论。"

> **Dev:** "04 记录表单是不是都要做成动态表单？"
> **Domain expert:** "不是，而且动态表单平台已于 2026-05-24 完全退役（见 ADR 0001）。所有记录表单应落到对应的独立业务 `*Record` 模型，不再使用动态表单引擎。"

> **Dev:** "鸡蛋房日报能不能做成一张 EggRoomShiftRecord 大表？"
> **Domain expert:** "不能。EggRoomShiftRecord 只能是工作台会话，库存、温湿度、清洁、消毒液浓度和计量校准要分别归账到真实业务对象。"

> **Dev:** "来料登记后是不是马上生成原辅料批次，检验只是补一张记录？"
> **Domain expert:** "不是。来料检验是入库门禁；最终合格或让步审批通过后，才生成可用原辅料批次和库存流水。"

> **Dev:** "水质余氯标准和今天测出来的 0.3ppm 能不能都放在 InspectionItem？"
> **Domain expert:** "不能。InspectionStandard/InspectionItem 是配置；今天这次检测是 InspectionRecord，0.3ppm 是 InspectionRecordItem 的实际值。"

> **Dev:** "文件名带 `-质检` 的表，能不能都合并成一张 QualityInspectionWorkbench 大表？"
> **Domain expert:** "不能。QualityInspectionWorkbench 只是质检入口；QualityInspectionTask 只是待办。中心温度、品评、净含量、氮气、虫害、水质这些真实结果都要落到 InspectionRecord/InspectionRecordItem 或对应专项对象。"

> **Dev:** "QualityInspectionTask 里直接放实测值和判定，会不会更方便？"
> **Domain expert:** "不行。任务只说明要检什么、谁检、什么时候检、完成状态和生成了哪条 InspectionRecord。实测值、判定、附件和不合格来源必须在 InspectionRecordItem。"

> **Dev:** "车间温湿度、冰柜温度、水质、虫害检查都能放 EnvironmentRecord 吗？"
> **Domain expert:** "不能。温湿度、压差、冰柜温度这种简单点位观察走 EnvironmentRecord；水质、虫害、卫生检查、环境微生物这种有检验项目和判定的，走 InspectionRecord/InspectionRecordItem。"

> **Dev:** "机头浆料比重、料温、烤炉过程参数能不能也走 EnvironmentRecord？"
> **Domain expert:** "不能。它们是生产批次或工序里的过程参数，走 ProcessMonitorRecord；EnvironmentRecord 只放点位环境观察。"

> **Dev:** "称油间、厕所、鸡蛋房这些日常清洁记录，要不要每个区域做一张固定表？"
> **Domain expert:** "不要。区域差异由 CleaningPlanTemplate、CleaningPlan 和 CleaningPlanItem 配置出来；现场只生成 CleaningRecord 和 CleaningRecordItem。新增区域时维护方案，不改代码。"

> **Dev:** "消毒浓度 200ppm 能不能直接放进 CleaningRecordItem？"
> **Domain expert:** "目标浓度来自 CleaningPlanItem；实际测出来的浓度走 SanitizerConcentrationCheck。CleaningRecordItem 可以引用这次浓度检查，但不要把浓度检查塞成普通清洁明细字段。"

> **Dev:** "不合格里能不能直接放原因、整改措施和验证结果，做成一张表？"
> **Domain expert:** "不能。不合格只记录问题事实和来源；整改措施用 CorrectiveAction，关闭验证用 VerificationRecord。"

> **Dev:** "导出的追溯 PDF、现场照片、作业指导书版本，能不能都放 Document？"
> **Domain expert:** "不能。作业指导书版本是 Document/DocumentVersion；现场照片是 EvidenceFile；某次追溯 PDF 是 EvidenceExport 产生的证据文件；报告版式是 ExportTemplate。"

> **Dev:** "首发证据导出是不是要把 282 张原始表单的纸面版式都做出来？"
> **Domain expert:** "不要。首发只做主链证据包，证明对象关系、状态、检验、异常、整改和追溯快照完整。原表版式复刻属于后续 ExportTemplate 治理。"

> **Dev:** "首发证据包里要不要做手写签名、CA 电子签章或者复杂签字区？"
> **Domain expert:** "不要。只冻结审批/审核快照：谁提交、谁审核/审批、审批时间、审批结论和当时审批意见。"

> **Dev:** "首发证据包下载要不要做 Excel，或者把所有附件打成 zip？"
> **Domain expert:** "不要。系统内权威是 JSON dataSnapshot；用户下载简单 PDF 或 HTML/PDF 摘要。附件只列文件名、链接、hash、上传人和上传时间，原始附件不打包进 zip。"

> **Dev:** "首发能不能从供应商、来料单、原料批次、检验记录或异常单直接导出证据包？"
> **Domain expert:** "不要。首发证据包唯一入口是产品批次；其他对象只能作为产品批次证据链中的节点出现。"

> **Dev:** "首发证据包必须包含发货单、客户和完整影响范围吗？"
> **Domain expert:** "不作为硬要求。首发预留下游影响范围区块；有可靠发货或召回关联就展示，没有也不能阻塞产品批次主链证据包。"

> **Dev:** "产品批次页面点导出时，要不要先让用户手动生成 TraceabilitySnapshot？"
> **Domain expert:** "不要。业务页面一键导出时系统自动先冻结新的追溯快照，再生成证据导出；高级追溯页可以从已有快照再导出。"

> **Dev:** "EvidenceExport.dataSnapshot 里只存对象 ID，打开时再查当前名称和状态，可以吗？"
> **Domain expert:** "不可以。证据数据快照要同时存对象 ID 和当时可读事实；旧证据包不能因为供应商改名、物料改名、检验标准换版而变样。"

> **Dev:** "同一个产品批次已经导出过证据包了，再点导出要覆盖旧的还是复用旧的？"
> **Domain expert:** "都不要。点“生成新证据包”就新建一条 TraceabilitySnapshot 和 EvidenceExport；历史列表里的“重新下载”只下载旧文件。"

> **Dev:** "产品批次还没完成，或者还有未关闭异常，能不能生成证据包？"
> **Domain expert:** "不能生成正式证据包。可以生成当前快照预览，标记 incomplete 并列出缺口，但不能当审核证据。"

> **Dev:** "正式证据包里的完成状态到底看 `completed`、`released_at`，还是另一个放行状态？"
> **Domain expert:** "首发只看 `ProductionBatch.status = completed` 作为批次完成门槛。`pending` 和 `in_progress` 只能预览，`cancelled` 永远不能导出正式证据包；`released_at / released_by` 有值就冻结进证据快照，但不作为首发硬门槛。"

> **Dev:** "`completed` 是包装确认、成品入库、质检完成，还是换下一批号时触发？"
> **Domain expert:** "`completed` 由批次关闭确认触发。连续生产时换下一批号会触发上一批关闭确认；非连续生产时由操作员在停机、结束生产或无下一批号时手动关闭当前批号。投料完成、生产运行结束或质检完成都不能单独把产品批次改成 completed。"

> **Dev:** "批次关闭确认要不要保存关闭人、关闭时间、关闭原因、实际产量或入库量快照？"
> **Domain expert:** "不要。批次关闭确认只是状态动作，不新增这些字段。入库量看成品入库的库存流水，证据包从库存流水里取入库事实。"

> **Dev:** "如果产品停了，但还没有成品入库库存流水，能不能先关闭批次？"
> **Domain expert:** "不能。没有成品入库库存流水，就没有可审计的入库量闭环，只能保持未完成或预览状态。"

> **Dev:** "批次已经 completed 后，发现漏入库，能不能继续追加普通成品入库流水？"
> **Domain expert:** "不能。completed 之后不能再当普通生产入库追加 `production_in`。漏入库或更正要走库存调整 `adjustment`，并指向这个产品批次。"

> **Dev:** "成品入库 `production_in` 是包装确认生成，还是成品入库确认生成？同一个产品批次能不能多次入库？"
> **Domain expert:** "由成品入库确认生成，而且确认人应是仓库或成品暂存区接收人。包装员工只提交待入库的包装事实，不直接写库存流水；只有仓库或成品暂存区确认接收后，才写 `InventoryMovement(object_type = production_batch, movement_type = production_in)`。同一个产品批次关闭前可以多次成品入库，每次入库确认各写一条 `production_in`，都指向同一个产品批次。"

> **Dev:** "既然 noidear 已经有仓库模块，成品入库确认要不要新建一个模块？"
> **Domain expert:** "不要新建平行模块。它归现有仓库模块，只是在库存流水能力里补产品批次入库：复用 `WarehouseModule`、`InventoryMovement` 和库存流水服务，新增或扩展一个记录 `production_batch + production_in` 的动作即可。"

> **Dev:** "成品入库确认入口放仓库模块，还是放产品批次详情？"
> **Domain expert:** "两个入口都可以有。仓库接收人从仓库任务入口处理最顺手，生产/追溯人员从产品批次详情查看也要能触发或查看。但底层只能走同一个仓库库存流水动作，不能两个页面各写一套入库逻辑。"

> **Dev:** "成品库存要不要单独建余额表？"
> **Domain expert:** "首发不建。先以产品批次相关库存流水查询或汇总为准，避免又多一个和库存流水抢写的数据源。以后如果性能或报表需要，再考虑由库存流水派生只读余额投影。"

> **Dev:** "成品入库确认时，仓库能不能修改产品批次号？"
> **Domain expert:** "不能。仓库只能选择或确认已有产品批次，不能在入库侧改批号。首发不做现场实物批号扫描、录入或自动比对；如果现场人工发现批号不一致，按线下流程退回生产/包装侧重新确认，或另行异常处理。"

> **Dev:** "如果目前拿不到可靠的包装事实数据，要不要新增 PackagingConfirmation 或 PackagingOutputRecord，再用包装累计数量限制入库数量？"
> **Domain expert:** "不要新增独立包装确认或包装产出记录。首发不做包装数量门禁，成品入库数量以仓库或成品暂存区接收确认为准。"

> **Dev:** "TraceabilitySnapshot 有了，能不能直接拿它当年度追溯演练或产品召回？"
> **Domain expert:** "不能。Snapshot 只是一次查询结果的冻结证据；Drill 是演练活动，Recall 是真实召回事件，它们都引用 Snapshot，但不能被 Snapshot 替代。"

> **Dev:** "培训统计页面能不能自己用 xlsx 生成 Excel？"
> **Domain expert:** "不行。Excel 导出由后端统一生成，前端只负责触发下载。"

> **Dev:** "培训统计导出遇到合同外的项目状态，要不要写成未知状态继续导出？"
> **Domain expert:** "不同意静默降级。培训状态超出合同是数据或代码合同损坏，导出应失败并提示修正状态，而不是写成未知或已取消。"

> **Dev:** "用户和文档要不要保留一个后台批量导入页面？"
> **Domain expert:** "不要。批量导入不放在产品功能里；需要初始化或大批量灌数据时，直接走开发/运维脚本。"

> **Dev:** "删掉批量导入后，要不要顺手建一个新的 CLI 导入脚本？"
> **Domain expert:** "本轮不要。删除产品功能就是边界，后续真需要批量导入再按新需求设计，不保留旧半成品，也不新增替代入口。"

> **Dev:** "代码里有 mobile.css 和 isMobile，是不是不能删除 mobile 工程？"
> **Domain expert:** "那是 Web/H5 主系统的响应式适配，不是移动端交付面；未设计的移动端交付面可以剔除。"

> **Dev:** "备份管理页面可以直接在后端 docker exec 容器做数据库和 MinIO 备份吗？"
> **Domain expert:** "不保留。备份属于部署运维基线，不放在业务 Web/H5 主系统里；MinIO 作为业务文件存储继续保留。"

> **Dev:** "删健康管理是不是也要删文档引用健康和访客健康状态？"
> **Domain expert:** "不是。删除的是应用内依赖健康/系统信息运维 API；文控引用健康和人员健康相关业务字段继续保留。"

> **Dev:** "删回收站是不是要把所有 deletedAt 字段都删掉？"
> **Domain expert:** "不是。删除的是跨模块通用回收站入口；各业务对象自己的归档、作废、软删除字段按业务规则保留。"

> **Dev:** "删批量导出是不是以后都不能导出了？"
> **Domain expert:** "不是。删除的是后台通用导出中心；真实业务场景里的统计、审计、追溯和 PDF 导出继续留在对应模块里。"

> **Dev:** "偏差报告导出现在依赖 ExportModule，那删通用导出中心是不是也删偏差报告导出？"
> **Domain expert:** "不是。删除 `/export/*` 通用入口和通用工具页；偏差报告导出作为偏差模块自己的业务导出保留，并迁出通用 ExportModule。"

> **Dev:** "`/export/task-records` 可以直接删吗？"
> **Domain expert:** "通用入口可以删，但记录填写结果必须能在记录管理里批量导出。新入口归记录模块，跨模板时返回 zip，每个模板一个 Excel。"

> **Dev:** "记录管理页点导出，是只导出当前页吗？"
> **Domain expert:** "不是。导出按当前筛选条件取全部结果，分页只影响页面展示，不影响导出范围。"

> **Dev:** "记录导出要不要先让用户选择字段？"
> **Domain expert:** "本轮不做字段选择 UI，默认导出全部字段。后端可以保留字段参数作为以后扩展，但前端不暴露。"

> **Dev:** "记录导出要不要做后台任务，或者允许一次导出所有历史记录？"
> **Domain expert:** "本轮先做同步导出，上限由后端控制为 10000 条。超过上限就让前端展示后端错误并提示用户缩小筛选范围，先不引入异步导出任务。"

> **Dev:** "状态不筛选时，记录导出要不要包含草稿？"
> **Domain expert:** "默认不包含草稿，因为草稿不是正式填写结果；用户明确筛选草稿时可以导出，用于内部排查。"

> **Dev:** "记录导出遇到附件、图片、签名和富文本，要不要把原始文件也打包进 zip？"
> **Domain expert:** "本轮不做完整证据包。Excel 里写文件名、链接、签名状态或纯文本；原始附件包以后按记录归档包/证据包单独设计。"

## Flagged ambiguities

- "生产批次"、"喷码批次"、"成品批次"曾被混用。已决议：业务文档统一使用 **产品批次**，代码当前由 `ProductionBatch` 承载。
- "计划批次"曾可能被误解为生产计划发布时提前生成的批次。已决议：**生产计划** 发布只派生 **现场执行任务**，不提前生成 **产品批次**；产品批次必须来自实际喷码、包装确认或成品入库确认。
- "`confirmProductBatch` / 产品批次确认"曾可能被误解为“创建批号、入库、完成批次”的一键动作。已决议：产品批次确认只表示“这个批号开始了”；**成品入库确认** 才写 `production_in`；**批次关闭确认** 才能把 `ProductionBatch.status` 变成 `completed`。
- "包装确认"曾可能被误解为成品库存入库动作。已决议：**包装确认** 只记录包装、喷码和数量等生产事实，不直接生成 `production_in`；**成品入库确认** 才生成成品入库 **库存流水**，确认人应是仓库或成品暂存区接收人；同一 **产品批次** 关闭前允许多次入库确认并产生多条 `production_in`。
- "入库改批号"曾可能被误解为仓库侧可在成品入库时修正产品批次号。已决议：**成品入库确认** 不得修改产品批次号，只能选择已有 **产品批次**；首发不新增现场实物批号扫描、录入或自动比对功能。若现场人工发现批号不一致，按线下流程退回生产/包装侧或另行异常处理。
- "包装产出记录"曾可能被建议为成品入库数量门禁来源。已决议：当前不新增独立 `PackagingConfirmation` / `PackagingOutputRecord`，也不做包装累计数量门禁；成品入库数量以仓库或成品暂存区接收确认为准。
- "`completed` 批次状态"曾可能被误解为投料完成、生产运行结束、质检完成或成品入库即自动完成。已决议：`ProductionBatch.status = completed` 只能由 **批次关闭确认** 触发；连续生产用换下一批号关闭上一批，非连续生产用手动关闭当前批号。关闭前必须已经存在该产品批次的成品入库 **库存流水**（`production_in`）；关闭确认不新增关闭人、关闭时间、关闭原因或数量快照；入库量以 **库存流水** 为准。`completed` 后不得继续追加普通 `production_in`，漏入库或更正只能走 `adjustment`。
- `ProductionRun` 曾可能被误解为生产计划或现场任务。已决议：**生产运行** 只是现场运行上下文，不是首发主链必经节点，不得用于补 `ProductionPlan`、`ProductionTask` 或 `ProductionBatch` 的缺口。
- `BatchMaterialUsage` 曾可能被误解为人工录入表。已决议：它是 **投料记录** 的追溯桥接结果，由 `MixingExecutionLine` 确认后自动生成，不提供人工创建页面或公开创建 API。
- `BatchMixingAggregation` 曾可能被误解为投料明细或库存流水。已决议：它只承接 **执行归集关系**，用于说明哪些配料/投料执行归到哪个 **产品批次**；不得替代 `BatchMaterialUsage` 的逐物料批次用量明细。
- `StockRecord` 和 `InventoryMovement` 曾可能形成两套库存动作流水。已决议：**库存流水** 的长期统一事实源是 `InventoryMovement`；**库存余额** 只表达当前数量/位置/状态。现有 `StockRecord` 若继续表达 `in/out/return/scrap` 动作，只能作为待收敛旧口径，不能与 `InventoryMovement` 双写两套权威流水。
- `Location`、`AreaPoint`、`WorkshopArea`、清洁点、温控点、采样点和虫害点曾可能被拆成多套位置对象。已决议：业务标准名统一为 **区域点位** / `AreaPoint`，noidear 当前落点为 `WorkshopArea`；专项点位对象只保存规则或记录，不重复保存物理位置身份。
- `Equipment`、`MeasuringEquipment`、`EquipmentAsset` 曾可能被理解为三套设备台账。已决议：**设备资产** 是业务标准名；noidear 当前用 `Equipment` 承接普通设备/设施维护，用 `MeasuringEquipment` 承接计量器具和校准，不另建平行设备台账。外检报告是 `EvidenceFile`，维护和校准分别走 `MaintenanceRecord` 与 `CalibrationRecord`。
- `InspectionStandard` / `InspectionItem` 曾可能被误解为检验执行记录。已决议：它们只承接 **检验标准** 和 **检验项目** 配置；现场实测值、判定、附件和不合格来源必须落到 `InspectionRecord` / `InspectionRecordItem`。
- `QualityInspectionWorkbench` / `QualityInspectionTask` 曾可能被误解为质检结果事实源。已决议：质检工作台只组织入口，质检任务只承接待办和完成关系；真实检验值、判定、报告、附件和异常来源必须落到 `InspectionRecord/InspectionRecordItem` 或明确专项对象。
- `EnvironmentRecord`、`ProcessMonitorRecord` 和 `InspectionRecord` 曾可能被混用。已决议：简单点位环境观察走 `EnvironmentRecord`；绑定生产批次或工序的过程参数走 `ProcessMonitorRecord`；需要检验模板、项目、判定或报告的水质、虫害、卫生、微生物、车辆卫生等走 `InspectionRecord/InspectionRecordItem`。
- `ProductProfile`、`ProductLabelProfile`、`ProductSpecification` 和 `ProductAllergenProfile` 曾可能被做成多张手填产品宽表。已决议：能手填的是 **产品档案**、**配方**、**物料过敏原画像**、**企业资质画像**、**产品风险区判定** 和 **产品验证记录** 等源头事实；标签画像、产品过敏原和规格书原则上是聚合/派生/导出快照，不得成为第二套事实源。
- `RetainedSample`、`RetainedSampleInspection` 和 `ShelfLifeStudy` 曾可能被合成留样大宽表。已决议：留样主对象只承接样品身份、来源、保存条件、数量、状态和处置；检查/复检是事件；保质期实验是多节点活动，节点实测值走 `InspectionRecord/InspectionRecordItem`。
- `Customer`、`ExternalParty`、`Supplier` 曾可能被混用。已决议：客户是 `ExternalParty(party_type=customer)`，承运商、废弃物收集方和外包服务方也归 `ExternalParty`；采购供应商仍归 `Supplier`，非供应商外部方评价走 `ExternalPartyEvaluation`。
- `VisitorRecord` 和 `AccessDeclaration` 曾可能互相抢字段。已决议：`VisitorRecord` 承接一次来访事件；`AccessDeclaration` 承接健康/保密/卫生声明、员工外出、货物/包裹放行等声明或放行事件，不把这些字段全部塞进访客登记。
- 日常清洁记录表曾可能按区域硬编码成多张表。已决议：区域差异用 `CleaningPlanTemplate`、`CleaningPlan`、`CleaningPlanItem` 配置；执行结果用 `CleaningRecord`、`CleaningRecordItem`；消毒液、余氯和浓度检查独立落到 `SanitizerConcentrationCheck`。
- `NonConformance` 曾可能被误解为异常整改大宽表。已决议：它只承接 **不合格** 事件本体和精确来源；原因分析、纠正预防措施和责任期限落到 `CorrectiveAction`；关闭验证和有效性证据落到 `VerificationRecord`。
- `Document` / `DocumentVersion` 曾可能被误解为所有文件和证据的统一落点。已决议：它们只承接受控文件和文件版本；业务附件落到 `EvidenceFile`，某次审核/追溯/记录导出落到 `EvidenceExport`，版式和字段绑定落到 `ExportTemplate`。
- "首发证据导出"曾可能被误解为 282 张源表版式全部复刻。已决议：首发只做 **首发主链证据包**，证明主链对象关系、状态流、检验结论、异常整改、追溯快照和关键附件索引完整；完整纸面表单版式归后续 `ExportTemplate` 治理。
- "签字"曾可能被误解为手写签名图片、CA 电子签章或复杂签字区排版。已决议：系统只承接 **审批/审核快照**，冻结提交人、审核/审批人、审批时间、审批结论和审批意见，不做签名图片和电子签章。
- "证据包下载"曾可能被误解为 Excel 主格式或附件 zip 包。已决议：系统内权威数据是 JSON 数据快照；用户下载简单 PDF 或 HTML/PDF **证据摘要文件**；附件只列索引，不打包全部原始文件。
- "证据包导出入口"曾可能被理解为任意对象都能导出。已决议：首发唯一根对象是 **产品批次**；供应商、来料单、原辅料批次、检验记录和异常单等对象只作为证据包内的链路节点，后续如需扩展根对象另行设计。
- "下游影响范围"曾可能被理解为首发必填发货/客户链。已决议：首发证据包只预留下游区块；已有可靠发货、客户或召回关联时展示，缺失时不阻塞首发主链验收。
- "导出证据包"曾可能被实现成先手动建快照、再手动导出，或每次打开文件时实时重查链路。已决议：业务页面一键导出自动生成新的 **追溯快照** 并绑定 **证据导出**；高级追溯页可从已有快照再次导出；导出后不得靠实时重查链路改变历史证据。
- "`dataSnapshot`" 曾可能被实现成只保存对象 ID 的关系图。已决议：**证据数据快照** 必须保存对象 ID 和当时可读事实；旧证据包打开、下载或重印时不得按当前主数据重新渲染关键事实。
- "重新导出"曾可能混用为覆盖旧证据包、复用旧快照或下载旧文件。已决议：按钮和接口语义拆开；**生成新证据包** 必须新建快照和导出，**重新下载** 只能读取历史导出文件。
- "未完成批次证据包"曾可能被做成正式导出。已决议：`pending`、`in_progress` 或 `cancelled` 批次，以及未关闭异常或主链事实缺失，只能生成 **当前快照预览**；**正式证据包** 必须满足 `ProductionBatch.status = completed` 和完整性门禁。`released_at / released_by` 只作为可冻结证据事实，不是首发硬门槛。
- `TraceabilitySnapshot` 曾可能被误解为追溯演练或召回事件。已决议：它只承接一次追溯查询结果的不可变快照；`TraceabilityDrill` 承接正追/反追演练活动；`ProductRecall` 承接真实召回事件并引用快照确定影响范围。
- `FinishedGoodsBatch` 曾被用作成品批次层。已决议：它不再作为核心业务批次概念，业务侧统一使用 **产品批次**。
- "删除产品"曾用于产品退出正常业务。已决议：用户侧统一使用 **产品归档**，历史追溯证据保留。
- "配方"和"Formula"不再并列。已决议：业务文档统一使用 **配方**。
- "体系文件库"和"文档台账"曾作为两个入口并存。已决议：统一收敛为 **体系文件中心** 的不同视图。
- "04 记录表单"曾被误解为普通文件夹或统一填报页。已决议：它是 **记录表单索引**，用于路由到业务模块；动态表单平台已于 2026-05-24 退役（见 ADR 0001），不再作为路由目标。
- `EggRoomShiftRecord`、`LaundryWorkRecord`、`ShiftHandoverCheck` 曾可能被误解为源表大表模型。已决议：这类对象只有在作为 **工作台会话** 或专项业务记录时成立；工作台会话不得保存库存、环境、清洁、浓度、计量或检验等事实字段，只保存会话元数据和真实业务对象引用。
- "来料检验"曾可能被理解为已有批次后的补充记录。已决议：**来料检验门禁** 在可用 **原辅料批次** 生成之前完成；不合格不生成可用批次，让步必须审批通过后才生成批次和库存流水。
- "Excel 导入导出"曾在前端页面各自生成或解析。已决议：产品功能保留业务导出，由后端生成；**应用内批量导入** 剔除，前端不得保留浏览器端 `xlsx` 依赖。
- "mobile"曾同时指移动工程、后端移动 API 和 Web 响应式样式。已决议：当前产品保留 **Web/H5 主系统**，剔除未设计的 **移动端交付面**，但不删除 Web 响应式样式。
- "备份管理"曾作为业务系统页面出现。已决议：剔除 **应用内备份管理**；数据库和对象存储备份由部署运维基线处理，业务系统保留 MinIO 文件存储能力。
- "健康管理"曾同时指运维健康 API、文控引用健康和人员健康状态。已决议：剔除 **应用内健康管理**，保留 **运行存活探针** 和业务健康语义。
- "回收站"曾作为跨业务对象统一恢复入口。已决议：剔除 **通用回收站**；不因此删除业务对象自己的 `deletedAt`、归档、作废或撤销语义。
- "批量导出"曾作为后台通用工具页和多个业务模块内导出混用。已决议：剔除 **通用批量导出中心**；保留 **业务模块导出**。偏差报告导出保留在偏差模块内，不再依赖通用 `ExportModule` 或 `/export/deviation-reports`。
- "任务记录导出"曾在通用导出中心里指向 `Record` 数据，命名不准确。已决议：改称 **记录填写结果导出**，由记录模块提供 `POST /records/export`；旧 `/export/task-records` 删除。
- "盘点差异"不得默认按普通盘盈盘亏跳过，也不默认把所有轻微差异升级成复杂整改。已决议：班后盘点由上一班交出，下一班用班前盘点承接确认；不单独建接班盘点。发现差异时先在盘点记录中保留差异和备注；影响生产或追溯时，再由负责人确认并转入异常处理。
- `approval`、`unified-approval`、`workflow`、`task`、`todo`、`record-task`、`scheduled-task` 曾被混用。已决议：业务口径区分 **正式审批**、**待办收件箱**、**记录填写任务**、**周期任务生成器** 和 **工作流编排**。
