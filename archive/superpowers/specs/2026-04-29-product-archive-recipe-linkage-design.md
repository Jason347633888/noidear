# 产品归档联动配方与工序设计

## 1. 背景

当前产品目录中的产品被移出正常业务后，配方管理仍可能继续显示该产品下的配方。用户侧看到的是“产品已经不在产品目录里，但配方还在正常列表中”，这会造成两个问题：

1. 业务语义不一致：产品已经退出正常业务，配方却仍像可用配方一样展示。
2. 食品安全风险：后续生产、开产、工序维护可能误用已退出业务的产品或配方。

根据主数据与追溯模型，`Product -> Recipe -> RecipeLine -> ProcessStep` 是产品主数据链的一部分。产品退出正常业务时，配方和工序也必须退出正常业务；但历史生产批次、投料、追溯证据不能被硬删除。

因此本设计统一使用“归档”作为用户侧概念，不再在产品目录和配方管理里使用“删除产品 / 删除配方”的业务文案。

## 2. 目标

1. 产品退出正常业务时，前台文案统一为“归档产品”。
2. 归档产品时，同步让该产品下的配方退出可用列表。
3. 归档产品时，同步让该产品下的工序退出正常工序列表或选择器。
4. 配方管理增加“可用配方 / 归档配方”两个视图。
5. 可用配方视图只显示产品仍有效且配方未归档的数据。
6. 归档配方视图可以查看已归档配方，并显示归档原因。
7. 保留历史生产批次、投料记录、追溯链和报表证据。

## 3. 不做范围

本阶段不做以下事情：

1. 不新增 `ArchivedProduct`、`ArchivedRecipe` 等归档专用表。
2. 不物理删除产品、配方、配方明细、工序。
3. 不把产品和配方纳入现有回收站模块。
4. 不新增归档恢复流程。
5. 不改生产批次、投料、追溯报表的数据结构。
6. 不重做产品目录、配方管理和工序管理的整体 UI。
7. 不处理全部历史脏数据的批量迁移脚本；数据修复可以后续单独出计划。

## 4. 可复用模块

### 4.1 Product 模块

现有 `ProductService` 已经维护产品主数据，并且当前“删除”行为本质上是给 `Product.deleted_at` 打标。这个机制可以复用为底层归档标记。

设计要求：

- 用户侧文案改为“归档”。
- 服务侧提供明确的 `archiveProduct` 或 `archive` 语义。
- 旧的删除入口如需兼容，可以内部转到归档逻辑，但不再作为新业务入口暴露给用户。

### 4.2 Recipe 模块

现有 `Recipe.status` 已支持 `draft / active / archived`。这正好承载配方归档，不需要新表。

设计要求：

- 配方归档时更新 `Recipe.status = archived`。
- 正常配方列表排除 `archived`。
- 归档配方列表包含 `archived`，以及产品已归档导致不可用的配方。

### 4.3 ProcessStep 模块

现有 `ProcessStep.deleted_at` 已支持工序软隐藏。产品归档时可以复用该字段，让工序退出正常列表和选择器。

设计要求：

- 产品归档时，同产品下未隐藏的 `ProcessStep` 统一打归档/隐藏标记。
- 正常工序列表继续只查 `deleted_at = null`。

### 4.4 配方管理页面

现有 `RecipeList.vue` 已有列表、产品筛选、状态标签和操作列。它适合扩展为两个视图：

- 可用配方
- 归档配方

不需要新增独立页面。

### 4.5 产品目录页面

现有 `ProductList.vue` 是用户触发产品退出正常业务的入口。这里应将操作按钮、确认弹窗和反馈文案从“删除”改为“归档”。

### 4.6 生产批次与生产开产模块

`ProductionBatchService` 和 `ProductionRunService` 已经要求产品 `active + deleted_at = null`、配方 `active` 才能建批次或开产。这个校验应继续作为后端保险。

设计要求：

- 不重复开发新的生产拦截逻辑。
- 产品归档和配方归档后，现有校验应自然拒绝新生产引用。

### 4.7 Document 归档范式

文档模块已有清晰的归档语义：归档不是删除，保留历史，可记录原因。产品/配方归档可以借鉴这种业务语言，但不复用文档服务代码。

### 4.8 RecycleBin 模块

回收站模块不适合复用。它服务的是“删除/恢复”语义，且当前只覆盖文档、记录模板和记录。产品与配方是主数据链，应该走业务归档，而不是回收站。

## 5. 业务规则

### 5.1 产品归档

产品归档表示产品退出正常业务，不表示历史数据被删除。

归档后：

1. 产品不再出现在正常产品选择器中。
2. 产品不再用于新建生产批次或生产开产。
3. 产品下的配方同步归档。
4. 产品下的工序退出正常工序列表。
5. 历史生产批次、投料记录、成品批次、发货、投诉、召回和追溯记录不变。

### 5.2 配方归档

配方归档表示该配方版本退出正常使用。

归档后：

1. 配方不再出现在可用配方列表。
2. 配方不再用于新建生产批次或生产开产。
3. 配方明细不删除，用于历史查看和追溯解释。
4. 已有关联生产批次仍保留 `recipeId / recipeName` 等历史引用。

### 5.3 工序归档

工序跟随产品退出正常业务。

归档后：

1. 工序不再出现在正常工序列表。
2. 工序不再作为新流程或新配置的可选项。
3. 历史过程记录不变。

## 6. 数据规则

| 对象 | 当前可复用字段 | 设计语义 |
| --- | --- | --- |
| Product | `deleted_at` | 产品已归档，退出正常业务 |
| Recipe | `status = archived` | 配方版本已归档，退出可用配方 |
| RecipeLine | 依附 `Recipe` | 不单独删除，保留历史配方结构 |
| ProcessStep | `deleted_at` | 工序已归档或隐藏 |
| ProductionBatch | 现有 `productId / recipeId / productName / recipeName` | 历史生产证据，不随产品归档改写 |
| BatchMaterialUsage | 现有 `productionBatchId / materialBatchId / recipeLineId` | 历史投料证据，不随产品归档改写 |

## 7. 查询规则

### 7.1 可用产品

可用产品应满足：

```text
Product.deleted_at = null
Product.status = active
```

### 7.2 可用配方

可用配方应满足：

```text
Recipe.status in (draft, active)
Product.deleted_at = null
Product.status = active
```

如果后续业务决定 `draft` 不应进入生产选择器，可以在生产选择器中继续收窄为 `Recipe.status = active`。配方管理中的“可用配方”可以展示 `draft`，但生产开产只能使用 `active`。

### 7.3 归档配方

归档配方应包含：

```text
Recipe.status = archived
OR Product.deleted_at is not null
OR Product.status != active
```

归档原因展示优先级：

1. 产品已归档
2. 产品已退出正常业务
3. 配方版本已归档

## 8. 用户体验

### 8.1 产品目录

产品目录操作列使用“归档”，不使用“删除”。

归档确认弹窗应表达：

```text
归档后产品、配方和工序将退出正常业务，历史追溯记录会保留。
```

成功反馈：

```text
归档成功
```

失败反馈：

```text
归档失败，请重试
```

### 8.2 配方管理

配方管理顶部增加视图切换：

- 可用配方
- 归档配方

可用配方视图：

- 显示正常产品下的 `draft / active` 配方。
- 操作允许“修改配方”和“归档”。

归档配方视图：

- 显示归档配方和产品已归档导致不可用的配方。
- 显示归档原因。
- 默认只读，不提供修改入口。

## 9. 接口设计

### 9.1 产品归档

建议新增接口：

```text
POST /products/:id/archive
```

服务逻辑：

1. 校验产品存在且属于当前公司。
2. 开启事务。
3. 标记 `Product.deleted_at = now()`。
4. 将该产品下未归档配方更新为 `Recipe.status = archived`。
5. 将该产品下未隐藏工序更新为 `ProcessStep.deleted_at = now()`。
6. 返回归档后的产品。

旧接口 `DELETE /products/:id` 如果需要兼容，可以内部转到同一归档逻辑。

### 9.2 配方归档

建议新增接口：

```text
POST /recipes/:id/archive
```

服务逻辑：

1. 校验配方存在且属于当前公司。
2. 更新 `Recipe.status = archived`。
3. 返回归档后的配方。

旧接口 `DELETE /recipes/:id` 如果需要兼容，可以内部转到同一归档逻辑。

### 9.3 配方列表

建议扩展现有接口：

```text
GET /recipes?archive=false
GET /recipes?archive=true
```

`archive=false` 返回可用配方。  
`archive=true` 返回归档配方。

按产品查询也应支持同样参数：

```text
GET /recipes/product/:productId?archive=false
GET /recipes/product/:productId?archive=true
```

## 10. 错误处理

1. 产品不存在或已不在正常查询范围内：返回“产品不存在或已归档”。
2. 配方不存在：返回“配方不存在”。
3. 归档过程中任一步失败：事务回滚，避免产品已归档但配方仍可用。
4. 生产或开产引用归档产品/配方：继续使用现有错误，“产品不存在或未启用”或“配方不存在、未激活或不属于该产品”。

## 11. 测试方案

### 11.1 后端单元测试

覆盖：

1. 产品归档会在事务内更新 Product、Recipe、ProcessStep。
2. 配方归档只更新 `Recipe.status = archived`，不硬删。
3. 可用配方列表排除产品已归档和配方已归档记录。
4. 归档配方列表包含产品已归档和配方已归档记录。

### 11.2 前端组件测试

覆盖：

1. 产品目录操作按钮显示“归档”。
2. 产品归档确认弹窗不出现“删除”和“不可撤销”。
3. 配方管理可切换“可用配方 / 归档配方”。
4. 归档配方视图显示归档原因，且不显示“修改配方”操作。

### 11.3 手工验证

路径：

1. 新建或选择一个产品。
2. 给该产品建立 active 配方和工序。
3. 在配方管理确认该配方出现在可用配方。
4. 在产品目录归档该产品。
5. 回到配方管理，确认该配方从可用配方消失。
6. 切换归档配方，确认该配方出现且原因为“产品已归档”。
7. 尝试新建生产批次或开产，确认不能引用该产品/配方。
8. 查看已有生产批次和追溯记录，确认历史证据不丢失。

## 12. 风险与约束

| 风险 | 处理 |
| --- | --- |
| 用户以为归档等于删除 | 全部前台文案使用“归档”，确认弹窗明确历史记录保留 |
| 产品归档后配方仍在可用列表 | 产品归档事务和配方列表过滤双保险 |
| 硬删破坏追溯链 | 本设计禁止物理删除产品、配方和配方明细 |
| 回收站语义混淆 | 产品/配方不接入 RecycleBin |
| 旧接口仍叫 DELETE | 保留兼容，但内部转归档；新页面和新 API 使用 archive |

## 13. 后续可选增强

以下不进入本阶段：

1. 产品归档原因、归档人、归档时间独立字段。
2. 配方归档原因、归档人、归档时间独立字段。
3. 归档产品恢复流程。
4. 归档配方导出。
5. 数据修复脚本：把当前已归档产品下仍 active 的配方统一归档。

## 14. 验收标准

1. 产品目录中不再出现“删除产品”文案，用户操作为“归档产品”。
2. 归档产品后，该产品从正常产品列表和选择器中退出。
3. 归档产品后，该产品下配方从可用配方列表消失。
4. 归档产品后，该产品下工序从正常工序列表消失。
5. 配方管理可以查看归档配方列表。
6. 归档配方列表能区分“产品已归档”和“配方版本已归档”。
7. 历史生产批次、投料记录、追溯记录不被删除或改写。
8. 新建生产批次和生产开产不能引用已归档产品或已归档配方。
